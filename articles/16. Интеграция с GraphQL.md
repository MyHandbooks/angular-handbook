GraphQL — это современный декларативный язык запросов и серверная среда для выполнения этих запросов на основе строго типизированной схемы данных. В отличие от традиционной архитектуры REST, предоставляющей множество эндпоинтов для фиксированных ресурсов, GraphQL предлагает единую точку входа (Entry Point), позволяя клиентскому приложению гибко и точно конструировать структуру запрашиваемого ответа.

---

## 1. Глубокое понимание концепции (Core Concept)

### Философия GraphQL и отличие от REST

Главное отличие GraphQL от REST заключается в переходе от ресурс-ориентированного подхода (где каждый URL представляет сущность) к **графовой модели данных**, где сущности связаны между собой ребрами графа.

GraphQL был разработан для решения трех ключевых проблем традиционного REST API:

1.  **Избыточное получение данных (Over-fetching):** REST-эндпоинт возвращает фиксированную структуру данных. Если компоненту нужно отобразить только имя пользователя, запрос к `/api/users/1` все равно вернет весь профиль с адресами, телефонами и историей заказов. Это приводит к лишней нагрузке на сеть и процессор при парсинге JSON. В GraphQL клиент запрашивает только конкретное поле: `user { name }`.
2.  **Недостаточное получение данных (Under-fetching):** Если для отрисовки страницы профиля нужны данные пользователя и список его последних постов, в REST часто приходится делать два последовательных сетевых запроса: сначала `/api/users/1`, затем `/api/users/1/posts`. Это увеличивает задержку (Latency). GraphQL позволяет получить связанные сущности за один сетевой запрос: `user { name posts { title } }`.
3.  **Сложность версионирования (API Versioning):** При изменении структуры данных в REST приходится создавать новые версии эндпоинтов (`/api/v1/`, `/api/v2/`). В GraphQL схема расширяется декларативно: новые поля добавляются безболезненно, а устаревшие помечаются директивой `@deprecated` без нарушения обратной совместимости.

#### Схема как контракт (Schema & Types)
Схема GraphQL жестко типизирована. Она является строгим контрактом между клиентом и сервером. В схеме объявляются скалярные типы (`Int`, `Float`, `String`, `Boolean`, `ID`), объектные типы (состоящие из набора полей), списки (`[Type]`) и модификатор обязательности (`!`, указывающий на то, что поле не может вернуть `null`).

---

### Основные операции и синтаксис GraphQL

Каждое взаимодействие клиента с GraphQL-сервером описывается одной из трех основных операций:

*   **Query (Запрос):** Чтение данных с сервера (аналог `GET` в REST). На транспортном уровне запросы обычно отправляются методом `POST` в теле HTTP-пакета.
*   **Mutation (Мутация):** Изменение данных на сервере (аналог `POST`, `PUT`, `DELETE` в REST). Мутации также могут возвращать измененные данные, что избавляет от необходимости совершать последующий повторный Query-запрос.
*   **Subscription (Подписка):** Постоянное соединение, инициируемое сервером для передачи событий в реальном времени (обычно работает поверх протокола WebSockets).

#### Динамика запроса: Переменные и Аргументы
GraphQL-поля могут принимать аргументы напрямую (например, `user(id: "123")`). Для построения переиспользуемых и динамических запросов используются переменные (`Variables`). Переменные объявляются в сигнатуре операции (например, `$userId: ID!`) и подставляются в аргументы полей.

---

### Интеграция Apollo Client в Angular

Для интеграции с GraphQL в экосистеме Angular стандартом де-факто является библиотека **Apollo Angular** (`apollo-angular`). Она предоставляет реактивные обертки над ядром `@apollo/client`, интегрируя сетевые запросы в RxJS-потоки.

Настройка в современной standalone-архитектуре производится в `app.config.ts` с помощью провайдера `provideApollo()`:
*   `HttpLink`: Связующий слой, настраивающий Apollo Client использовать стандартный `HttpClient` для отправки сетевых запросов.
*   `InMemoryCache`: Интеллектуальный слой кэширования Apollo Client. Он автоматически нормализует данные ответов по уникальным полям `id` или `_id`, предотвращая лишние сетевые запросы при повторном обращении к тем же сущностям.

## 2. Ментальная модель

Сравнение сетевой нагрузки и логики запроса: REST vs GraphQL

```text
1. REST API (Under-fetching - требуется 2 запроса):
   Клиент                                                 Сервер REST
     │                                                         │
     ├─────── [Запрос 1]: GET /api/users/1 ───────────────────►│ (Парсинг БД)
     │◄────── [Ответ 1]: { id: 1, name: "Иван" } ──────────────┤
     │                                                         │
     ├─────── [Запрос 2]: GET /api/users/1/posts ─────────────►│ (Поиск по индексу)
     │◄────── [Ответ 2]: [ { id: 101, title: "RxJS..." } ] ────┤

2. GraphQL (Точечный сбор за 1 запрос):
   Клиент                                                 Сервер GraphQL
     │                                                         │
     ├─────── [Запрос]: POST /graphql ────────────────────────►│ (Разрешение графа)
     │        query { user(id: 1) { name posts { title } } }   │  - Вызов Resolver User
     │                                                         │  - Вызов Resolver Posts
     │◄────── [Ответ]: {                                       │
     │          "data": {                                      │
     │            "user": {                                    │
     │              "name": "Иван",                            │
     │              "posts": [ { "title": "RxJS..." } ]        │
     │            }                                            │
     │          }                                              │
     │        } ───────────────────────────────────────────────┤
```

## 3. Практические примеры кода

### Плохой пример: Строковая интерполяция и ручные запросы

Ручное склеивание GraphQL-запроса через строковую интерполяцию переменных приводит к синтаксическим ошибкам и делает невозможной статическую валидацию запросов компилятором.

```typescript
import { Component, inject, OnInit } from '@angular/core';
import { HttpClient } from '@angular/common/http';

@Component({
  selector: 'app-bad-graphql',
  template: `<h1>Категории</h1>`,
  standalone: true
})
export class BadGraphqlComponent implements OnInit {
  private http = inject(HttpClient);

  ngOnInit() {
    const parentId = 'category-123';
    
    // Плохо: Прямой запрос через HttpClient без использования Apollo
    // Плохо: Ручная интерполяция переменной parentId в тело GraphQL-запроса
    const badQuery = {
      query: `
        query GetCategories {
          categories(parentId: "${parentId}") {
            id
            name
          }
        }
      `
    };

    this.http.post('/graphql', badQuery).subscribe(res => {
      console.log('Результат:', res);
    });
  }
}
```

### Хороший пример: Использование Apollo Client и строго типизированных запросов

Использование тега `gql` для парсинга абстрактного синтаксического дерева (AST) запроса, строго типизированных переменных, Apollo-сервиса и современной отписки через `takeUntilDestroyed`.

```typescript
import { Component, inject, DestroyRef } from '@angular/core';
import { takeUntilDestroyed } from '@angular/core/rxjs-interop';
import { Apollo, gql } from 'apollo-angular';
import { Observable } from 'rxjs';
import { map } from 'rxjs/operators';

interface Category {
  id: string;
  name: string;
}

interface CategoriesResponse {
  categories: Category[];
}

interface CategoriesVariables {
  parentId: string;
}

// Хорошо: Использование тега gql для объявления статически валидируемого запроса
const GET_CATEGORIES = gql`
  query GetCategories($parentId: ID!) {
    categories(parentId: $parentId) {
      id
      name
    }
  }
`;

@Component({
  selector: 'app-good-graphql',
  template: `
    @if (categories$ | async; as list) {
      <ul>
        @for (cat of list; track cat.id) {
          <li>{{ cat.name }}</li>
        }
      </ul>
    }
  `,
  standalone: true
})
export class GoodGraphqlComponent {
  private apollo = inject(Apollo);
  private destroyRef = inject(DestroyRef);

  // Хорошо: Декларативный типизированный реактивный поток данных
  readonly categories$: Observable<Category[]> = this.apollo
    .query<CategoriesResponse, CategoriesVariables>({
      query: GET_CATEGORIES,
      variables: {
        parentId: 'category-123' // Хорошо: Передача переменных через специальный объект
      }
    })
    .pipe(
      takeUntilDestroyed(this.destroyRef), // Безопасное завершение стрима при уничтожении компонента
      map(result => result.data.categories) // Извлечение чистых данных из структуры ответа
    );
}
```

## 4. Вопросы для собеседования

### Вопрос 1: Каким образом GraphQL решает проблемы "Over-fetching" (избыточного получения данных) и "Under-fetching" (недостаточного получения данных)?
**Ответ:** 
*   **Over-fetching:** GraphQL решает эту проблему за счет концепции декларативного выбора полей. Клиент описывает форму ответа в теле запроса (Selection Set). Сервер возвращает ровно те ключи, которые были запрошены, предотвращая отправку по сети неиспользуемых байтов данных.
*   **Under-fetching:** GraphQL решает эту проблему благодаря графовой структуре типов. Сервер предоставляет резолверы для связей между объектами. Клиент может запросить сущность и все связанные с ней дочерние коллекции любого уровня вложенности в рамках одного сетевого запроса, устраняя необходимость совершать каскадные HTTP-запросы.

### Вопрос 2: В чем разница между передачей параметров в GraphQL-запрос напрямую (In-line arguments) и использованием механизма переменных (Variables)?
**Ответ:** 
*   **In-line аргументы** жестко прописываются в строке запроса (например, `user(id: "123")`). Это делает запрос негибким, так как для поиска другого пользователя придется заново генерировать саму строку запроса (что ломает статическое кэширование AST на клиенте и сервере).
*   **Variables (Переменные)** отделяют статический шаблон запроса от динамических данных. Запрос компилируется один раз с помощью тега `gql`, а значения подставляются отдельно на этапе выполнения. Это позволяет GraphQL-серверам кэшировать синтаксическое дерево запроса (Query Plan) и повышает безопасность, предотвращая инъекции вредоносного кода.

### Вопрос 3: Как работает нормализация кэша в Apollo Client (`InMemoryCache`) и по какому признаку она по умолчанию объединяет данные?
**Ответ:** 
`InMemoryCache` деструктурирует ответы от GraphQL-сервера и сохраняет их в плоской структуре данных в виде ключ-значение. По умолчанию в качестве уникального ключа объекта кэш использует комбинацию системного имени типа `__typename` и уникального идентификатора объекта `id` (или `_id`), например: `Category:category-123`. 
Если два разных запроса возвращают один и тот же объект с одинаковым `id`, Apollo Client автоматически обновляет это поле в кэше. Все компоненты, подписанные на этот объект, мгновенно увидят актуальные данные без необходимости повторного обращения к серверу.

### Вопрос 4: Зачем при конфигурации Apollo Client в Angular (`provideApollo`) внедряется класс `HttpLink`? Какую роль он играет?
**Ответ:** 
Ядро Apollo Client (`@apollo/client`) является платформонезависимым и по умолчанию использует для сетевых запросов стандартный браузерный `fetch`. 
Класс `HttpLink` из пакета `apollo-angular/http` связывает Apollo Client со стандартным модулем `HttpClient` в Angular. Это крайне важно, так как позволяет GraphQL-запросам проходить через стандартную цепочку функциональных перехватчиков (Interceptors), использовать механизмы проксирования (`proxy.conf.json`), ложиться в инфраструктуру SSR (Server-Side Rendering) и тестироваться с помощью стандартных тестовых утилит Angular.
