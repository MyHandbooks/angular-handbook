TypeScript является строго типизированным компилируемым надмножеством языка JavaScript. Он решает фундаментальную проблему динамической типизации JavaScript, позволяя выявлять логические и синтаксические ошибки на этапе статического анализа (компиляции), а не во время исполнения (runtime) приложения.

---

## 1. Глубокое понимание концепции (Core Concept)

### Явная и неявная типизация. Вывод типов
*   **Явная типизация (Explicit Typing)**: Разработчик вручную указывает принадлежность переменной или аргумента к конкретному типу данных. Это улучшает читаемость кода для распределенных команд и исключает двусмысленность.
*   **Неявная типизация (Implicit Typing / Type Inference)**: Компилятор автоматически выводит тип переменной на основе присвоенного ей значения (литерала). Например, объявление `let count = 10` неявно привязывает переменную `count` к типу `number`.

### Система типов TypeScript под капотом

#### 1. any vs unknown (Верхние типы / Top Types)
*   **any**: Отключает проверки компилятора для переменной, делая ее совместимой со всеми типами в обе стороны. Использование `any` нивелирует преимущества статической типизации и скрывает потенциальные runtime-ошибки.
*   **unknown**: Безопасный аналог `any`. Переменной типа `unknown` можно присвоить любое значение, но совершать над ней операции (вызывать методы, передавать аргументом) запрещено до тех пор, пока ее тип не будет явно сужен (Type Narrowing) с помощью логических проверок.

#### 2. void, null, undefined
*   **void**: Сигнализирует об отсутствии возвращаемого значения у функции или метода. В JavaScript подобный тип отсутствует, а функции без явного `return` возвращают значение `undefined`.
*   **null и undefined**: Обозначают отсутствие данных. При включенном строгом флаге компиляции `--strictNullChecks` типы `null` и `undefined` перестают быть совместимыми со всеми остальными типами (например, нельзя присвоить `null` переменной типа `string`). Для объявления возможности отсутствия значения используется символ опциональности `?` или объединение (Union) с `null`.

#### 3. Enum (Перечисления)
Конструкция `enum` группирует набор именованных констант. По умолчанию компилятор присваивает числовые индексы (0, 1, 2...), но также поддерживаются строковые перечисления. В отличие от большинства возможностей TypeScript, `enum` компилируется в реальный JavaScript-объект, поддерживающий реверсивное сопоставление (для числовых значений).

#### 4. Union и Intersection (Объединение и Пересечение)
*   **Union (`|`)**: Логическое "ИЛИ". Значение переменной может соответствовать одному из указанных типов. До сужения типа доступны только те свойства и методы, которые присутствуют во всех объединенных типах одновременно.
*   **Intersection (`&`)**: Логическое "И". Объединяет свойства нескольких типов в один единый тип. Чаще всего применяется к объектам для масштабирования их сигнатур.

#### 5. Литеральные типы
Позволяют использовать конкретные значения (строковые литералы, числа, логические значения) в качестве типов. С их помощью создаются жестко ограниченные диапазоны допустимых значений (например, `type Status = 'pending' | 'success' | 'rejected'`).

## 2. Ментальная модель

Тип `unknown` можно представить как "запечатанный контейнер". Чтобы извлечь его содержимое и безопасно выполнить операции, необходимо пропустить значение через логический фильтр (Type Guard/Сужение типа).

```text
  [ Любое входящее значение ]
              │
              ▼
    ┌───────────────────┐
    │  Тип: unknown     │ (Значение запечатано, вызов методов заблокирован)
    └─────────┬─────────┘
              │
              ▼
   /─────────────────────\
  <  Проверка typeof?     > (Логический фильтр / Type Guard)
   \─────────────────────/
      │               │
      │ Да            │ Нет
      ▼               ▼
┌─────────────┐   ┌─────────────────────────────────────┐
│  string     │   │ Неизвестное состояние               │
│ (Доступны   │   │ (Вызов методов по-прежнему закрыт)  │
│  методы     │   └─────────────────────────────────────┘
│  строки)    │
└─────────────┘
```

## 3. Практические примеры кода

### Плохой пример: Использование `any` и отсутствие строгих проверок

Код уязвим к падению в runtime из-за отсутствия валидации динамических данных и небезопасного приведения типов.

```typescript
// Unsafe API data fetcher
function processApiResponse(payload: any) {
  // Компилятор не выдает ошибку, но код упадет в runtime, если payload.user undefined
  const email = payload.user.email;
  console.log(email.toLowerCase()); 
}

// Избыточное числовое перечисление, заменяемое литеральным типом
enum TrafficLightLegacy {
  Red,
  Yellow,
  Green
}

function handleLight(light: TrafficLightLegacy) {
  if (light === TrafficLightLegacy.Red) {
    console.log('Stop');
  }
}
```

### Хороший пример: Использование `unknown`, Type Guard и литеральных типов

Код безопасен, защищен от неожиданных API-ответов компилятором на этапе разработки.

```typescript
type UserData = {
  user: {
    email: string;
  }
};

// Функция-проверка (User-defined Type Guard) для безопасного сужения типа
function isUserData(obj: unknown): obj is UserData {
  return (
    typeof obj === 'object' &&
    obj !== null &&
    'user' in obj &&
    typeof (obj as any).user === 'object' &&
    'email' in (obj as any).user &&
    typeof (obj as any).user.email === 'string'
  );
}

function processSafeApiResponse(payload: unknown): void {
  if (isUserData(payload)) {
    // Внутри этого блока payload автоматически приведен к типу UserData
    const email = payload.user.email;
    console.log(email.toLowerCase());
  } else {
    console.error('Ошибка: Некорректный формат данных ответа API');
  }
}

// Литеральное объединение вместо тяжеловесного числового enum
type TrafficLight = 'red' | 'yellow' | 'green';

function handleSafeLight(light: TrafficLight): void {
  if (light === 'red') {
    console.log('Stop');
  }
}
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем заключается разница между `any` и `unknown` с точки зрения системы типов?
**Ответ:** 
Оба типа являются "верхними типами" (Top Types), способными хранить значения любого типа. Различие состоит в правилах проверки типов:
*   Тип `any` отключает типобезопасность. Вы можете читать любые свойства, вызывать функции и присваивать переменную типа `any` переменным других строго заданных типов без явного преобразования.
*   Тип `unknown` заставляет соблюдать типобезопасность. Вы не можете вызывать методы или читать свойства переменной типа `unknown`, пока компилятор не убедится в ее истинном типе посредством сужения (например, через блоки `typeof`, `instanceof` или пользовательские защитники типов — Type Guards).

### Вопрос 2: Как компилируется числовой `enum` в JavaScript и к каким потенциальным багам runtime это может привести?
**Ответ:** 
Числовой `enum` компилируется в IIFE-функцию (самовызывающуюся функцию), которая создает объект с двунаправленным связыванием (reverse mapping). Например:
```javascript
var Color;
(function (Color) {
    Color[Color["Red"] = 0] = "Red";
})(Color || (Color = {}));
```
Это приводит к тому, что объект `Color` содержит свойства `Color.Red === 0` и `Color[0] === "Red"`. 
**Потенциальный баг:** С точки зрения системы типов TypeScript, числовой `enum` совместим с любым числом (`number`). То есть функция, принимающая `Color`, без ошибок компиляции примет аргумент `999`, что может привести к сбою логики на сервере или в пользовательском интерфейсе во время выполнения. Для надежного контроля значений рекомендуется использовать объединение литеральных типов (Union of literal types).

### Вопрос 3: Как работает флаг компиляции `--strictNullChecks` и как он изменяет поведение типов `null` и `undefined`?
**Ответ:** 
Без флага `--strictNullChecks` значения `null` и `undefined` являются подтипами всех остальных типов (string, number, boolean) и могут быть присвоены любой переменной без явных предупреждений компилятора. Это часто приводит к ошибкам вроде `Cannot read properties of null`.
При включении `--strictNullChecks` типы `null` и `undefined` изолируются. Они могут быть присвоены только переменным своего собственного типа, типу `any` или `unknown`. Если переменная типа `string` может содержать отсутствие значения, разработчик обязан явно объявить это через Union: `string | null`.

### Вопрос 4: Объясните разницу работы операторов Union (`|`) и Intersection (`&`) при применении их к объектным типам данных.
**Ответ:** 
*   **Union (`TypeA | TypeB`)**: Создает тип, экземпляр которого может соответствовать либо `TypeA`, либо `TypeB`. До выполнения сужения типа компилятор разрешает безопасный доступ **только к тем свойствам, которые объявлены в обоих объектах одновременно (пересечение их интерфейсов)**.
*   **Intersection (`TypeA & TypeB`)**: Объединяет свойства типов. Экземпляр такого типа должен **строго содержать все обязательные свойства как из `TypeA`, так и из `TypeB` одновременно**.
