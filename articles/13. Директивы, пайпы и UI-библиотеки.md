Директивы и пайпы — это фундаментальные инструменты Angular, предназначенные для декларативного расширения возможностей HTML, управления поведением DOM-элементов и форматирования вывода данных непосредственно в шаблонах. Использование встроенных и кастомных декларативных инструментов в связке с экосистемой Angular Material позволяет проектировать масштабируемые и производительные интерфейсы уровня Enterprise.

## 1. Глубокое понимание концепции (Core Concept)

### Директивы в Angular

Директива — это класс с декоратором `@Directive`, который позволяет добавлять новое поведение или изменять структуру и внешний вид элементов в DOM. Существует три типа директив:

1.  **Компоненты (Components):** Директивы с собственным HTML-шаблоном.
2.  **Структурные директивы (Structural Directives):** Изменяют структуру DOM-дерева путем добавления, удаления или изменения элементов. В синтаксисе шаблонов выделяются префиксом `*` (например, `*ngIf`, `*ngFor` в устаревшем синтаксисе). Современный Angular использует новый управляющий синтаксис шаблонов (`@if`, `@for`, `@switch`), который компилируется в эффективные блочные структуры без накладных расходов на директивы.
3.  **Атрибутные директивы (Attribute Directives):** Изменяют внешний вид или поведение уже существующего DOM-элемента, компонента или другой директивы. Примеры встроенных директив: `NgClass` (динамические стили классов), `NgStyle` (инлайн-стили), `NgModel` (двустороннее связывание данных для форм).

#### Создание кастомных атрибутных директив
Для безопасного взаимодействия с DOM внутри директивы используются:
*   `ElementRef`: Предоставляет доступ к базовому DOM-элементу (через свойство `nativeElement`). Прямое изменение элемента через него не рекомендуется из-за рисков безопасности (XSS) и невозможности выполнения приложения в окружениях без DOM (например, при Server-Side Rendering / SSR).
*   `Renderer2`: Специальный сервисный слой (абстракция над DOM) для безопасной модификации стилей, атрибутов и классов элемента на любых платформах.
*   `@HostListener`: Декоратор, позволяющий декларативно подписываться на события DOM-элемента (хоста), на котором применена директива.
*   `@HostBinding`: Декоратор для связывания свойств хост-элемента со свойствами класса директивы.

### Механизм работы Пайпов (Pipes) под капотом

Пайпы — это специальные классы с декоратором `@Pipe`, реализующие интерфейс `PipeTransform`. Они выполняют декларативную трансформацию данных в шаблонах по бинарному принципу: `значение | названиеПайпа:аргумент1:аргумент2`.

#### Pure Pipes (Чистые пайпы — по умолчанию `pure: true`)
Чистый пайп вызывается механизмом Change Detection **только тогда**, когда изменяются входные параметры пайпа:
*   Для примитивных типов (`string`, `number`, `boolean`, `symbol`): при изменении самого значения.
*   Для ссылочных типов (`Object`, `Array`, `Date`): при изменении ссылки на объект.

Angular кэширует результат выполнения чистого пайпа (реализуя паттерн мемоизации). Если ссылка на объект внутри шаблона не изменилась, функция `transform()` не перезапускается, что дает колоссальный прирост производительности.

#### Impure Pipes (Нечистые пайпы — `pure: false`)
Нечистый пайп вызывается механизмом Change Detection **при каждом цикле проверки**, независимо от того, изменились ли входные данные. 
*   **Опасность:** Если внутри метода `transform` нечистого пайпа выполняются ресурсоемкие операции (фильтрация больших массивов, тяжелые вычисления), приложение начнет испытывать серьезные проблемы с производительностью (зависание UI, просадка FPS).
*   **Исключение:** `AsyncPipe` является нечистым пайпом по умолчанию. Он подписывается на `Observable` или `Promise`, вызывает Change Detection при поступлении новых асинхронных данных и автоматически отписывается при уничтожении компонента для предотвращения утечек памяти.

### UI-библиотеки и Angular Material

**Angular Material** — официальная библиотека UI-компонентов от Google, реализующая концепцию Material Design. Она построена на базе **Component Dev Kit (CDK)**.
*   **Angular CDK** — набор headless-примитивов (компонентов без стилей), решающих сложные инженерные задачи: управление фокусом, доступность (A11y), создание порталов и всплывающих окон (Overlay), виртуальный скроллинг и Drag and Drop. CDK позволяет создавать уникальные дизайн-системы с нуля, не тратя время на базовую логику доступности и позиционирования.

## 2. Ментальная модель

Различие в вызовах вычислений при Change Detection:

```text
Цикл проверки изменений (Change Detection) в шаблоне компонента
  │
  ├──► Обнаружен чистый пайп: {{ user | ageCalculator }}
  │     │
  │     └──► Входное значение (User) изменило ссылку?
  │            ├── Да ──► Выполнить transform() ──► Обновить UI
  │            └── Нет ─► Взять закэшированное значение (Пропуск)
  │
  ├──► Обнаружен нечистый пайп / вызов метода класса: {{ getAge(user) }}
  │     │
  │     └──► Вызывать функцию ВСЕГДА на каждом цикле CD (Даже при движении мыши/скролле)
  │            └──► Выполнить вычисления ──► Обновить UI (Высокая нагрузка)
```

## 3. Практические примеры кода

### Плохой пример: Вызов тяжелых методов класса в шаблоне

Вызовы методов класса непосредственно в интерполяции шаблона выполняются на каждом цикле проверки изменений, что полностью блокирует поток рендеринга при работе с большими наборами данных.

```typescript
import { Component } from '@angular/core';

interface User {
  name: string;
  salary: number;
}

@Component({
  selector: 'app-bad-pipes',
  standalone: true,
  template: `
    <ul>
      @for (user of users; track user.name) {
        <!-- Плохо: Метод calculateTax() будет выполняться десятки раз на любой чих в приложении -->
        <li>{{ user.name }} - Налог: {{ calculateTax(user.salary) | currency }}</li>
      }
    </ul>
  `
})
export class BadPipesComponent {
  users: User[] = [
    { name: 'Иван', salary: 100000 },
    { name: 'Мария', salary: 150000 }
  ];

  calculateTax(salary: number): number {
    console.log('Тяжелые вычисления налога запущены!'); // Лог забьет консоль
    // Имитация ресурсоемкой калькуляции
    return salary * 0.13;
  }
}
```

### Хороший пример: Использование кастомного чистого пайпа и кастомной директивы ховера

Использование чистого пайпа кэширует вычисления налога, а кастомная директива безопасно управляет стилями элемента через `Renderer2` и `@HostListener`.

```typescript
import { Pipe, PipeTransform, Directive, ElementRef, Renderer2, HostListener, Component } from '@angular/core';
import { CurrencyPipe } from '@angular/common';

// 1. Создаем производительный Чистый Пайп
@Pipe({
  name: 'taxCalculator',
  pure: true, // По умолчанию true. Метод transform запустится только при изменении примитива salary
  standalone: true
})
export class TaxCalculatorPipe implements PipeTransform {
  transform(salary: number, taxRate: number = 0.13): number {
    console.log('Чистый пайп: Вычисление налога выполнено только при изменении salary!');
    return salary * taxRate;
  }
}

// 2. Создаем кастомную директиву для безопасного интерактивного ховера
@Directive({
  selector: '[appHighlight]',
  standalone: true
})
export class HighlightDirective {
  constructor(
    private el: ElementRef,
    private renderer: Renderer2 // Безопасная абстракция над DOM для SSR/платформ
  ) {}

  @HostListener('mouseenter') 
  onMouseEnter(): void {
    this.changeBgColor('#f0f0f0');
  }

  @HostListener('mouseleave') 
  onMouseLeave(): void {
    this.changeBgColor('transparent');
  }

  private changeBgColor(color: string): void {
    this.renderer.setStyle(this.el.nativeElement, 'background-color', color);
  }
}

// 3. Собираем в компоненте
@Component({
  selector: 'app-good-pipes',
  standalone: true,
  imports: [TaxCalculatorPipe, HighlightDirective, CurrencyPipe],
  template: `
    <ul>
      @for (user of users; track user.name) {
        <!-- Хорошо: Данные мемоизированы. Директива управляет стилями без лишнего триггера CD для всего списка -->
        <li appHighlight>
          {{ user.name }} - Налог: {{ user.salary | taxCalculator:0.13 | currency }}
        </li>
      }
    </ul>
  `
})
export class GoodPipesComponent {
  users = [
    { name: 'Иван', salary: 100000 },
    { name: 'Мария', salary: 150000 }
  ];
}
```

## 4. Вопросы для собеседования

### Вопрос 1: Почему вызов метода класса компонента в интерполяции шаблона `{{ getTax() }}` считается антипаттерном производительности и как чистые пайпы решают эту проблему?
**Ответ:** 
При любой проверке изменений (Change Detection) в Angular, фреймворк обязан убедиться, что отображаемые в шаблоне данные актуальны. Так как Angular не может знать, изменился ли результат выполнения метода `getTax()` (метод может зависеть от внешнего изменяемого состояния), он вынужден вызывать этот метод при *каждом* проходе Change Detection. 
Чистые пайпы (`pure: true`) решают эту проблему за счет концепции мемоизации. Angular отслеживает только входные аргументы пайпа. Если ссылки на входные аргументы пайпа не изменились с момента предыдущей проверки, Angular полностью пропускает вызов метода `transform()` пайпа и подставляет ранее вычисленное и сохраненное значение, экономя процессорное время.

### Вопрос 2: В чем разница между Pure (чистыми) и Impure (нечистыми) пайпами? Приведите пример стандартного нечистого пайпа в Angular и объясните его необходимость.
**Ответ:** 
*   **Pure Pipe** вызывается только тогда, когда меняется ссылка на входные данные (для объектов/массивов) или само значение (для примитивов). Если мы мутируем массив (например, делаем `array.push(item)` без изменения ссылки на сам массив), чистый пайп не сработает.
*   **Impure Pipe** (с флагом `pure: false`) вызывается на каждом цикле Change Detection, реагируя на любые изменения в приложении, включая мутации объектов и массивов «изнутри».
*   **Пример:** `AsyncPipe` является нечистым пайпом по умолчанию. Он подписывается на входящий стрим `Observable` или `Promise`. Специфика асинхронности требует отслеживания поступления новых данных на каждом цикле проверки, а также управления подпиской (вызов `.unsubscribe()` при уничтожении компонента), что невозможно реализовать в рамках идеологии только чистых ссылочных проверок.

### Вопрос 3: Почему при написании кастомных директив для модификации DOM рекомендуется использовать `Renderer2`, а не напрямую изменять свойства через `ElementRef.nativeElement`?
**Ответ:** 
Прямое взаимодействие с DOM через `ElementRef.nativeElement` жестко связывает код приложения с браузерным окружением (платформой). Это приводит к следующим проблемам:
1.  **Нарушение работы SSR (Server-Side Rendering):** При рендеринге на сервере (например, с Angular Universal / SSR) браузерного DOM-дерева не существует. Обращение к `nativeElement` вызовет ошибку и падение серверного процесса сборки.
2.  **Безопасность:** Прямое внедрение скриптов или HTML через свойства элемента в обход Angular API повышает уязвимость приложения перед XSS-атаками.
3.  **Архитектура:** Использование `Renderer2` предоставляет абстрактный слой над платформой. `Renderer2` безопасно выполнит код в браузере, а на сервере преобразует эти вызовы в безопасные строковые представления без падения приложения.

### Вопрос 4: Какую роль играет Angular CDK при создании сложных корпоративных UI-компонентов и чем он отличается от Angular Material?
**Ответ:** 
*   **Angular Material** — это готовая библиотека UI-компонентов, жестко реализующая спецификацию Material Design со своими предустановленными стилями, анимациями и поведением.
*   **Angular CDK (Component Dev Kit)** — это библиотека абстрактных низкоуровневых утилит и "безголовых" (headless) заготовок без стилей. CDK решает сложные инфраструктурные задачи:
    *   `Overlay` (динамические всплывающие окна, контекстные меню, дропдауны с расчетом координат).
    *   `A11y` (доступность для скринридеров, утилита FocusTrap).
    *   `Drag and Drop` (перетаскивание списков с физикой).
    *   `Virtual Scroll` (рендеринг списков на миллионы строк).
CDK используется, когда требования к дизайну приложения не соответствуют стилям Material Design, но разработчикам необходимо быстро создать кастомные, доступные и сложные интерактивные компоненты с нуля.
