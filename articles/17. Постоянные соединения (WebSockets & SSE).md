Традиционный протокол HTTP работает по транзакционной схеме «запрос-ответ»: клиент инициирует соединение, получает данные, после чего сессия закрывается. Для интерактивных систем реального времени (чаты, финансовые котировки, дашборды) такой подход неэффективен. Постоянные соединения позволяют серверу отправлять данные клиенту (Push) в режиме реального времени, минимизируя накладные расходы на установление сессий.

## 1. Глубокое понимание концепции (Core Concept)

### Эволюция подходов к постоянным соединениям

Для преодоления ограничений классического HTTP были разработаны различные архитектурные подходы:

1.  **Long Polling (Длинный опрос):** Клиент отправляет стандартный HTTP-запрос, а сервер намеренно удерживает соединение открытым до тех пор, пока не появятся новые данные или не истечет тайм-аут. Получив данные, клиент немедленно отправляет новый запрос, возобновляя цикл.
    *   *Проблема:* При высокой частоте обновлений нагрузка на сеть напоминает «пилу». Каждый цикл сопровождается отправкой избыточных HTTP-заголовков, cookies и прохождением процедур авторизации.
2.  **Server-Sent Events (SSE):** Стандартизированный протокол односторонней передачи данных от сервера к клиенту, работающий поверх классического HTTP. Клиент открывает соединение с помощью встроенного класса `EventSource`, а сервер держит его открытым, отправляя поток текстовых событий.
    *   *Особенности:* Автоматическое переподключение при обрыве связи, поддержка кастомных типов событий, работа через стандартные порты `80` / `443`.
3.  **WebSockets:** Двунаправленный (полнодуплексный) протокол передачи данных, работающий по собственной схеме поверх TCP-соединения (протоколы `ws://` и защищенный `wss://`). Соединение устанавливается один раз через процедуру рукопожатия (Handshake) и позволяет обеим сторонам обмениваться сообщениями в любое время.

---

### Сравнение протоколов: WebSockets vs Server-Sent Events

При проектировании real-time слоя выбор технологии зависит от специфики задач:

| Критерий | WebSockets | Server-Sent Events (SSE) |
| :--- | :--- | :--- |
| **Направление передачи** | Двунаправленное (Full-Duplex) | Однонаправленное (Server-to-Client) |
| **Протокол** | Собственный протокол (`ws` / `wss`) | Стандартный HTTP |
| **Формат данных** | Текстовые и бинарные фреймы (Blob, ArrayBuffer) | Только текстовый формат (`text/event-stream`) |
| **Максимальное число соединений** | Лимитируется ресурсами сервера | В HTTP/1.1 ограничено браузером (6 соединений на домен) |
| **Обход прокси и файрволов** | Может блокироваться старыми прокси-серверами | Проходит без проблем (обычный HTTP-трафик) |
| **Восстановление соединения** | Требуется ручная реализация в коде | Встроено на уровне браузера (автореконнект) |

### Реализация WebSocket и SSE в Angular с помощью RxJS

Прямое использование нативного браузерного API в компонентах Angular нарушает декларативный реактивный подход и затрудняет тестирование. Интеграция с RxJS позволяет обернуть постоянные соединения в управляемые потоки.

#### Работа с SSE через EventSource
Браузерный класс `EventSource` транслирует события через стандартные колбэки. В Angular их оборачивают в `Observable` с помощью функции `fromEvent`. Важно вручную вызывать метод `.close()` у экземпляра `EventSource` при уничтожении подписчика, чтобы избежать утечек сетевых соединений.

#### Работа с WebSockets через WebSocketSubject
Пакет `rxjs/webSocket` предоставляет специализированный класс `WebSocketSubject`. Это реактивная обертка над нативным веб-сокетом, которая является одновременно и `Observable`, и `Observer`.
*   Чтение данных: подписка на `WebSocketSubject` запускает физическое соединение и транслирует входящие сообщения.
*   Отправка данных: вызов метода `.next(payload)` сериализует данные в JSON и отправляет их на сервер.
*   Управление жизненным циклом: при отписке последнего подписчика `WebSocketSubject` автоматически закрывает сетевое соединение.

## 2. Ментальная модель

Сравнение процесса установления соединения и передачи данных:

```text
1. Протокол Server-Sent Events (SSE):
   Клиент                                                 Сервер (HTTP)
     │                                                         │
     ├─ [1. Запрос соединения] ───────────────────────────────►│ Content-Type: text/event-stream
     │◄─ [2. Ответ 200 OK (Соединение открыто)] ───────────────┤ (Keep-Alive)
     │                                                         │
     │◄─ [3. Push-событие: data: "Событие 1"] ─────────────────┤ (Соединение удерживается)
     │◄─ [4. Push-событие: data: "Событие 2"] ─────────────────┤ (Односторонняя передача)


2. Протокол WebSockets:
   Клиент                                                 Сервер (WS)
     │                                                         │
     ├─ [1. Handshake]: GET /ws (HTTP Upgrade) ───────────────►│ Upgrade: websocket
     │◄─ [2. Handshake]: HTTP 101 Switching Protocols ─────────┤ (Соединение переключено на TCP)
     │                                                         │
     │◄─── [3. Бинаправленный обмен фреймами (Full-Duplex)] ──►│
     │     Клиент шлет данные ◄───► Сервер шлет данные         │
```

## 3. Практические примеры кода

### Плохой пример: Неуправляемое создание нативного WebSocket в компоненте

Прямая инициализация `new WebSocket` внутри компонента приводит к утечкам памяти (при уничтожении компонента соединение остается открытым), отсутствию автореконнекта и трудностям в обработке ошибок.

```typescript
import { Component, OnInit, OnDestroy } from '@angular/core';

@Component({
  selector: 'app-bad-ws',
  template: `<h1>Чат</h1>`,
  standalone: true
})
export class BadWsComponent implements OnInit, OnDestroy {
  private socket!: WebSocket;

  ngOnInit(): void {
    // Плохо: Использование сырого браузерного API вне сервисного слоя
    this.socket = new WebSocket('ws://localhost:8080');

    this.socket.onopen = () => {
      this.socket.send('Привет от клиента!');
    };

    this.socket.onmessage = (event) => {
      console.log('Новое сообщение:', event.data);
    };

    this.socket.onerror = (error) => {
      console.error('Ошибка сети:', error);
    };
  }

  ngOnDestroy(): void {
    // Плохо: Нет явного закрытия соединения при уничтожении компонента, сокет зависнет в памяти
  }
}
```

### Хороший пример: Реактивные сервисы постоянных соединений

Создание декларативных сервисов для SSE и WebSockets. Использование RxJS обеспечивает автоматическое закрытие соединений и интеграцию в общий реактивный цикл приложения.

#### 1. Сервис для работы с Server-Sent Events (SSE)
```typescript
// sse.service.ts
import { Injectable, OnDestroy, NgZone } from '@angular/core';
import { Observable, fromEvent } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class SseService implements OnDestroy {
  private eventSource: EventSource | null = null;

  constructor(private zone: NgZone) {}

  connect(url: string): Observable<MessageEvent> {
    return new Observable<MessageEvent>(observer => {
      // Инициализируем EventSource внутри NgZone.runOutsideAngular, чтобы частые PUSH-события 
      // не триггерили глобальный Change Detection (Zone.js) без необходимости
      this.zone.runOutsideAngular(() => {
        this.eventSource = new EventSource(url);

        this.eventSource.onmessage = (event: MessageEvent) => {
          this.zone.run(() => observer.next(event)); // Передаем данные обратно в зону Angular
        };

        this.eventSource.onerror = (error) => {
          this.zone.run(() => observer.error(error));
        };
      });

      // Хорошо: Функция очистки (Tear-down) закрывает соединение при отписке
      return () => {
        if (this.eventSource) {
          this.eventSource.close();
          this.eventSource = null;
        }
      };
    });
  }

  ngOnDestroy(): void {
    if (this.eventSource) {
      this.eventSource.close();
    }
  }
}
```

#### 2. Сервис для работы с WebSockets на базе RxJS
```typescript
// ws.service.ts
import { Injectable, OnDestroy } from '@angular/core';
import { webSocket, WebSocketSubject } from 'rxjs/webSocket';
import { Observable, EMPTY } from 'rxjs';
import { catchError } from 'rxjs/operators';

interface ChatMessage {
  user: string;
  text: string;
}

@Injectable({ providedIn: 'root' })
export class WsService implements OnDestroy {
  // Хорошо: Использование WebSocketSubject автоматически управляет сериализацией/десериализацией JSON
  private socket$: WebSocketSubject<ChatMessage> | null = null;

  connect(url: string): Observable<ChatMessage> {
    if (!this.socket$) {
      this.socket$ = webSocket<ChatMessage>({
        url,
        // Опционально: можно настроить логику реконнекта при закрытии соединения
        closingObserver: {
          next: () => console.log('Соединение WebSocket закрывается...')
        }
      });
    }

    return this.socket$.asObservable().pipe(
      catchError(error => {
        console.error('Ошибка WebSocket соединения:', error);
        return EMPTY; // Безопасное завершение потока при критической ошибке
      })
    );
  }

  send(message: ChatMessage): void {
    if (this.socket$) {
      this.socket$.next(message); // Отправка сообщения на сервер в формате JSON
    }
  }

  close(): void {
    if (this.socket$) {
      this.socket$.complete(); // Корректное закрытие сокета
      this.socket$ = null;
    }
  }

  ngOnDestroy(): void {
    this.close();
  }
}
```

## 4. Вопросы для собеседования

### Вопрос 1: В каких архитектурных сценариях технология Server-Sent Events (SSE) превосходит WebSockets и наоборот?
**Ответ:** 
*   **SSE превосходит WebSockets**, когда приложению требуется исключительно односторонняя доставка данных от сервера к клиенту (например, лента новостей, уведомления, статусы выполнения тяжелых задач на бэкенде, биржевые котировки). SSE работает поверх стандартного HTTP, автоматически восстанавливает соединение при обрывах (реконнект), поддерживает кастомные типы событий из коробки и легко масштабируется через стандартные балансировщики нагрузки.
*   **WebSockets превосходит SSE**, когда требуется двунаправленный обмен данными с минимальной задержкой (например, многопользовательские игры, real-time чаты, совместное редактирование документов). Также WebSockets необходим, если требуется передавать бинарные данные (файлы, стриминг аудио/видео), так как SSE поддерживает исключительно текстовый формат UTF-8.

### Вопрос 2: Как устроен процесс переключения протокола (Handshake) в WebSockets? Почему он начинается с обычного HTTP-запроса?
**Ответ:** 
Для установления WebSocket-соединения клиент отправляет стандартный HTTP GET-запрос на сервер, содержащий специальные заголовки:
*   `Upgrade: websocket` — указывает на желание переключить протокол.
*   `Connection: Upgrade` — сигнализирует промежуточным прокси о переключении.
*   `Sec-WebSocket-Key` — случайный ключ для проверки безопасности.

Этот процесс начинается с HTTP, чтобы гарантировать совместимость со стандартной сетевой инфраструктурой. Запрос проходит через те же порты (`80` или `443`), обрабатывается теми же веб-серверами и прокси. Если сервер поддерживает WebSockets, он возвращает ответ со статус-кодом `101 Switching Protocols`. После этого TCP-соединение остается открытым, а логика взаимодействия переключается с HTTP-пакетов на легковесные фреймы протокола WebSocket.

### Вопрос 3: С какими ограничениями безопасности и авторизации сталкиваются разработчики при использовании стандартного браузерного `EventSource` для SSE и как их обойти?
**Ответ:** 
Стандартный браузерный класс `EventSource` при инициализации (`new EventSource(url)`) не позволяет передавать кастомные HTTP-заголовки (например, заголовок `Authorization: Bearer <token>`), что делает невозможной классическую Bearer-авторизацию для закрытых эндпоинтов.
**Способы решения:**
1.  **Передача токена через Query-параметры:** Токен передается в URL (например, `/sse?token\=jwt_token`). Способ прост, но небезопасен, так как токены могут логироваться на промежуточных прокси-серверах.
2.  **Использование Cookies:** Если бэкенд и фронтенд находятся на одном домене (или настроен CORS), авторизационные куки прикрепляются к запросу `EventSource` автоматически (при включении флага `withCredentials: true`).
3.  **Использование сторонних библиотек-полифилов:** Например, `fetch-event-source` (от Microsoft). Она заменяет стандартный `EventSource` под капотом на API `fetch`, сохраняя поток `text/event-stream`, но позволяя передавать любые кастомные заголовки, включая `Authorization`.

### Вопрос 4: Как состояние `readyState` нативного WebSocket сопоставляется с жизненным циклом RxJS-потока `WebSocketSubject`?
**Ответ:** 
Свойство `readyState` нативного сокета может принимать четыре значения: `0` (CONNECTING), `1` (OPEN), `2` (CLOSING), `3` (CLOSED).
В RxJS-классе `WebSocketSubject` это сопоставляется со следующими фазами:
*   При создании экземпляра `webSocket()` сокет находится в состоянии покоя. Физическое соединение не устанавливается.
*   При первой подписке (`.subscribe()`) нативный сокет переходит в состояние `0` (CONNECTING), а затем в `1` (OPEN). Поток начинает транслировать сообщения.
*   При отправке данных через `.next()` сокет должен находиться в состоянии `1` (OPEN). Если соединение еще не установилось (`0`), `WebSocketSubject` буферизирует отправляемые сообщения и доставляет их сразу после открытия канала.
*   При отписке (`unsubscribe`) последнего подписчика или вызове `.complete()` сокет переходит в состояние `2` (CLOSING), а затем в `3` (CLOSED). Сетевой ресурс высвобождается автоматически.
