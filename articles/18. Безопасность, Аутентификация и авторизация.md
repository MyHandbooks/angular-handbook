Обеспечение безопасности веб-приложения строится на четком разделении процессов установления личности пользователя и проверки его прав на выполнение действий. В архитектуре современных одностраничных приложений (SPA) эти задачи решаются на стыке протоколов без сохранения состояния (Stateless), криптографических токенов и реактивных механизмов контроля переходов внутри клиентского фреймворка.

## 1. Глубокое понимание концепции (Core Concept)

### Различие между Аутентификацией и Авторизацией. Основные подходы

Безопасность любой информационной системы опирается на две последовательные фазы:

1.  **Аутентификация (Authentication):** Процесс проверки подлинности субъекта. Система сопоставляет предоставленные данные (пароль, токен, биометрия) с эталоном в базе данных, подтверждая личность пользователя. Результатом успешной аутентификации является выдача «удостоверения» (сессии или токена).
2.  **Авторизация (Authorization):** Процесс проверки прав доступа успешно аутентифицированного пользователя к конкретному ресурсу или операции (например, разрешено ли пользователю с ролью `user` удалять комментарии).

#### Архитектурные модели аутентификации

*   **Session-based (На основе сессий):** Традиционная модель с сохранением состояния (Stateful). При входе пользователя сервер создает запись сессии в своей памяти или базе данных, а клиенту возвращает уникальный идентификатор сессии (`session_id`) в HTTP-куке. При каждом запросе браузер автоматически отправляет куку, а сервер валидирует её по своей базе.
    *   *Проблема:* Сложность горизонтального масштабирования бэкенда (требуется репликация сессий между серверами или использование централизованных хранилищ типа Redis).
*   **Token-based (На основе токенов):** Модель без сохранения состояния (Stateless). После успешного входа сервер генерирует криптографический токен (обычно JSON Web Token / JWT) и передает его клиенту. Клиент сохраняет токен локально и отправляет его в заголовке каждого HTTP-запроса (`Authorization: Bearer <token>`). Серверу не нужно хранить состояние сессии — он проверяет валидность токена математически с помощью ключа шифрования.

#### Структура JWT (JSON Web Token)
Токен JWT состоит из трех частей, разделенных точками (`Header.Payload.Signature`):
1.  **Header (Заголовок):** Содержит тип токена (JWT) и алгоритм шифрования подписи (например, HS256).
2.  **Payload (Полезная нагрузка):** Набор утверждений (Claims) о пользователе (его ID, имя, роли, время истечения токена `exp`). Эти данные не шифруются, а лишь кодируются в Base64, поэтому их может прочитать любая сторона.
3.  **Signature (Подпись):** Хэш, вычисленный от закодированных Header и Payload с использованием секретного ключа сервера. Гарантирует, что данные в Payload не были изменены клиентом в процессе передачи.

### Протоколы авторизации и SSO (Single Sign-On)

Для интеграции безопасности на межплатформенном уровне используются специализированные стандарты:

*   **OAuth 2.0:** Протокол авторизации, позволяющий стороннему приложению получить ограниченный доступ к ресурсам пользователя на другом сервисе без передачи пароля (например, авторизация на сайте через Google-аккаунт).
*   **OpenID Connect (OIDC):** Протокол аутентификации, надстроенный поверх OAuth 2.0. Он вводит дополнительный стандартизированный `ID Token` (в формате JWT) для безопасной передачи профиля пользователя клиентскому приложению.
*   **SSO (Single Sign-On):** Метод единого входа, позволяющий пользователю авторизоваться один раз и получать доступ ко множеству независимых корпоративных систем без повторного ввода учетных данных. Реализуется через обмен токенами безопасности между провайдером идентификации (Identity Provider / IdP) и сервисными провайдерами (Service Providers / SP).

### Реализация авторизации в Angular

В Angular-приложениях реактивное управление сессией и правами доступа строится на базе трех ключевых инструментов:

1.  **Хранение состояния (Auth State):** Хранение JWT-токена и профиля текущего пользователя осуществляется внутри синглтон-сервиса (`AuthService`). В современном Angular состояние сессии декларативно описывается с помощью сигналов (`WritableSignal`), что позволяет шаблонам и компонентам мгновенно реагировать на вход/выход пользователя без ручных подписок.
2.  **Автоматическая подстановка токена (Interceptor):** Функциональный перехватчик `HttpInterceptorFn` клонирует каждый исходящий `HttpRequest` и внедряет заголовок `Authorization: Bearer <token>`, освобождая другие сервисы от необходимости вручную настраивать заголовки запросов.
3.  **Защита роутов (Guards):** Функциональный гард `CanActivateFn` проверяет наличие токена или необходимых ролей у пользователя перед переходом по адресу. Если прав недостаточно, гард прерывает навигацию и перенаправляет пользователя на страницу входа с помощью `Router.createUrlTree()`.

## 2. Ментальная модель

Схема токенной авторизации (Stateless TBA) в Angular:

```text
 Клиент (Браузер)                                                 Сервер
        │                                                           │
 [Форма входа] ── 1. POST /login (username, password) ─────────────►│ (Проверка пароля)
        │                                                           │  Генерация JWT
        │◄─────── 2. Ответ 200 OK { token: "header.payload.sig" } ──┤
        │
  [AuthService] ──► Сохраняет токен, обновляет Signal(user)
        │
  [Маршрутизация] ──► Переход на /dashboard
        │
  [AuthGuard] ──► Проверяет Signal(user) -> Разрешено!
        │
  [Запрос данных] ──► HttpClient.get('/api/data')
        │
  [HttpInterceptor] ──► Клонирует запрос, добавляет:
        │               Authorization: Bearer header.payload.sig
        │
        ├──────── 3. GET /api/data (с заголовком Bearer) ──────────►│ (Валидация подписи JWT)
        │                                                           │  Разрешено!
        │◄─────── 4. Ответ 200 OK [Данные приборной панели] ────────┤
```

## 3. Практические примеры кода

### Плохой пример: Классовый гард, ручная подписка и небезопасная передача заголовков

Использование устаревшего классового гарда засоряет DI-дерево. Ручное внедрение заголовков авторизации в каждом методе сервиса нарушает принцип DRY (Don't Repeat Yourself), а отсутствие централизованного реактивного состояния усложняет отслеживание статуса авторизации.

```typescript
// bad-auth.guard.ts
import { Injectable } from '@angular/core';
import { CanActivate, Router } from '@angular/router';

@Injectable({ providedIn: 'root' })
export class BadAuthGuard implements CanActivate {
  constructor(private router: Router) {}

  // Плохо: Устаревший классовый подход
  canActivate(): boolean {
    const token = localStorage.getItem('token');
    if (!token) {
      this.router.navigate(['/login']); // Ручной императивный редирект
      return false;
    }
    return true;
  }
}

// bad-data.service.ts
import { Injectable, inject } from '@angular/core';
import { HttpClient, HttpHeaders } from '@angular/common/http';

@Injectable({ providedIn: 'root' })
export class BadDataService {
  private http = inject(HttpClient);

  getData() {
    const token = localStorage.getItem('token');
    // Плохо: Ручная сборка заголовков для каждого сетевого запроса
    const headers = new HttpHeaders().set('Authorization', `Bearer ${token}`);
    return this.http.get('/api/data', { headers });
  }
}
```

### Хороший пример: Сигналы, функциональный гард и функциональный интерцептор

Современное декларативное решение: состояние пользователя обернуто в Signal, токен подставляется автоматически через функциональный интерцептор, а роуты защищены лаконичным функциональным гардом.

```typescript
// auth.service.ts
import { Injectable, signal, computed } from '@angular/core';

export interface UserProfile {
  id: string;
  name: string;
  role: string;
}

@Injectable({ providedIn: 'root' })
export class AuthService {
  // Хорошо: Хранение реактивного состояния авторизации через сигналы
  private currentUserSignal = signal<UserProfile | null>(null);
  
  readonly currentUser = this.currentUserSignal.asReadonly();
  readonly isAuthenticated = computed(() => this.currentUserSignal() !== null);

  login(token: string, profile: UserProfile): void {
    localStorage.setItem('auth_token', token);
    this.currentUserSignal.set(profile);
  }

  logout(): void {
    localStorage.clear();
    this.currentUserSignal.set(null);
  }

  getToken(): string | null {
    return localStorage.getItem('auth_token');
  }
}
```

```typescript
// jwt.interceptor.ts
import { HttpInterceptorFn } from '@angular/common/http';
import { inject } from '@angular/core';
import { AuthService } from './auth.service';

// Хорошо: Функциональный перехватчик автоматически прикрепляет Bearer-токен
export const jwtInterceptor: HttpInterceptorFn = (req, next) => {
  const authService = inject(AuthService);
  const token = authService.getToken();

  if (token) {
    const authReq = req.clone({
      headers: req.headers.set('Authorization', `Bearer ${token}`)
    });
    return next(authReq);
  }

  return next(req);
};
```

```typescript
// auth.guard.ts
import { CanActivateFn, Router } from '@angular/router';
import { inject } from '@angular/core';
import { AuthService } from './auth.service';

// Хорошо: Легковесный функциональный гард, возвращающий UrlTree в случае отказа
export const authGuard: CanActivateFn = (route, state) => {
  const authService = inject(AuthService);
  const router = inject(Router);

  if (authService.isAuthenticated()) {
    return true;
  }

  // Хорошо: Декларативное предотвращение перехода с генерацией дерева редиректа
  return router.createUrlTree(['/login']);
};
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем принципиальная разница между сессионной (Session-based) и токенной (Token-based) аутентификацией? Каковы их плюсы и минусы при масштабировании SPA?
**Ответ:** 
*   **Session-based (Stateful):** Состояние хранится на сервере. При входе создается сессия в памяти/БД бэкенда, а клиент идентифицируется по `session_id` в куках.
    *   *Плюсы:* Легко мгновенно аннулировать сессию со стороны сервера (например, принудительный логаут администратором). Меньше рисков утечки данных, так как токен авторизации не хранится в JS-доступной памяти.
    *   *Минусы:* Плохо масштабируется горизонтально. При наличии множества серверов бэкенда за балансировщиком нагрузки требуется синхронизация хранилища сессий (Redis) или настройка Sticky Sessions.
*   **Token-based (Stateless):** Состояние хранится на клиенте в виде JWT-токена. Сервер проверяет его математически по сигнатуре.
    *   *Плюсы:* Идеально масштабируется. Бэкенд-сервера абсолютно независимы, им не нужна общая база сессий для валидации запросов. Подходит для микросервисов.
    *   *Минусы:* Токен невозможно мгновенно аннулировать со стороны сервера до истечения его времени жизни (`exp`), если только не реализовывать черные списки токенов (Blacklists), что частично возвращает систему к Stateful-модели.

### Вопрос 2: Как устроен токен JWT? Почему клиентское приложение может прочитать данные внутри Payload, но не может их подделать?
**Ответ:** 
Токен JWT состоит из трех частей, закодированных в Base64 и разделенных точками: `Header.Payload.Signature`.
*   **Header** содержит метаданные (алгоритм шифрования подписи).
*   **Payload** содержит данные (Claims) о пользователе и времени жизни токена. Любой желающий может декодировать строку Payload из Base64 и прочитать ее содержимое, поэтому туда нельзя помещать конфиденциальные данные (например, пароли).
*   **Signature** формируется сервером. Сервер берет закодированные Header и Payload, конкатенирует их и шифрует с помощью секретного приватного ключа, который хранится исключительно на сервере.
Клиент не может подделать данные (например, изменить свою роль с `user` на `admin`), потому что при любом изменении в Payload изменится итоговый хэш подписи. При получении запроса сервер заново вычисляет подпись от пришедших Header и Payload с помощью своего секретного ключа. Если вычисленная сервером подпись не совпадет с подписью из токена, запрос будет немедленно заблокирован с ошибкой `401 Unauthorized`.

### Вопрос 3: Где безопаснее хранить JWT-токены в SPA-приложениях: в LocalStorage/SessionStorage или в HttpOnly Cookies? С какими уязвимостями связан этот выбор?
**Ответ:** 
Выбор места хранения — это компромисс между двумя типами уязвимостей:
1.  **LocalStorage / SessionStorage:**
    *   *Плюс:* Удобно использовать. Доступно из JS, токен легко извлекается и передается интерцептором в заголовке `Authorization`.
    *   *Минус (Критический):* Уязвимо к **XSS (Cross-Site Scripting)**. Если злоумышленник сможет внедрить сторонний JS-скрипт на страницу (через уязвимую форму, комментарии или стороннюю библиотеку), он сможет прочитать LocalStorage и украсть токен.
2.  **HttpOnly Cookies:**
    *   *Плюс:* Надежная защита от XSS. Заголовок `HttpOnly` полностью блокирует доступ к куке из JavaScript-кода. Скрипт злоумышленника не сможет ее прочитать.
    *   *Минус:* Уязвимо к **CSRF (Cross-Site Request Forgery)**. Так как браузер автоматически прикрепляет куки к запросам на целевой домен, злоумышленник может обманом заставить пользователя совершить вредоносный запрос со стороннего сайта. Для защиты требуется обязательное внедрение CSRF-токенов (`X-XSRF-TOKEN`) и заголовка `SameSite: Strict` / `Lax` для куки.

### Вопрос 4: Как реализовать механизм обновления токена доступа (Refresh Token) в Angular с помощью функционального перехватчика при получении ошибки `401 Unauthorized`?
**Ответ:** 
Реализация строится на перехвате ошибки внутри интерцептора с помощью RxJS-оператора `catchError`. Схема действий:
1.  Интерцептор ловит ошибку со статусом `401 Unauthorized`.
2.  Если процесс обновления токена еще не запущен, интерцептор блокирует выполнение текущего и последующих запросов, выставляет флаг `isRefreshing \= true` и отправляет запрос на эндпоинт обновления `/api/auth/refresh`, передавая сохраненный ранее `refresh_token`.
3.  Остальные запросы, приходящие в этот момент, ставятся в очередь ожидания с помощью RxJS `BehaviorSubject`, который хранит состояние готовности нового токена.
4.  При успешном получении нового `access_token` от сервера, мы сохраняем его, обновляем `BehaviorSubject`, сбрасываем флаг `isRefreshing` и повторно отправляем все скопившиеся в очереди запросы с новым токеном.
5.  Если запрос на обновление токена завершился неудачей, мы принудительно разлогиниваем пользователя (`authService.logout()`) и перенаправляем на страницу авторизации.
