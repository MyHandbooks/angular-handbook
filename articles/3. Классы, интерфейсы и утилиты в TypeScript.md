TypeScript расширяет спецификацию ES6-классов, добавляя механизмы строгого контроля доступа, поддержки интерфейсов и абстрактных классов. Использование объектно-ориентированных паттернов проектирования и встроенных утилит типизации позволяет создавать поддерживаемую, расширяемую и чистую архитектуру приложений в соответствии с принципами SOLID.

---

## 1. Глубокое понимание концепции (Core Concept)

### Модификаторы доступа и видимости членов класса
TypeScript предоставляет три модификатора доступа, которые регулируют видимость свойств и методов:
1.  **`public`** (значение по умолчанию): свойства и методы доступны из любого места программы (внутри класса, в классах-наследниках и во внешнем коде через экземпляр класса).
2.  **`private`**: члены класса доступны исключительно внутри самого класса. Попытка обращения к ним снаружи или из классов-наследников вызывает ошибку компиляции.
3.  **`protected`**: члены класса доступны внутри самого класса и во всех его наследниках, но остаются закрытыми для внешнего вызывающего кода.

Важно понимать, что модификаторы доступа TypeScript проверяются исключительно на этапе компиляции. После компиляции в JavaScript модификаторы удаляются, и свойства становятся обычными публичными полями объекта. Для обеспечения изоляции во время выполнения (runtime) в JavaScript следует использовать нативные приватные поля с префиксом `#`.

#### Модификатор `readonly`
Позволяет пометить поле объекта как "только для чтения" (аналог `const` для полей объекта). Инициализация таких полей возможна только в месте их объявления или внутри конструктора класса. Если значение не присвоено, поле по умолчанию принимает значение `undefined`.

---

### Интерфейсы и абстрактные классы

#### Интерфейс (`interface`)
Интерфейс — это виртуальная синтаксическая конструкция, предназначенная исключительно для описания структуры объекта (API) без предоставления деталей реализации. 
*   Все члены интерфейса неявно считаются публичными (`public`).
*   Интерфейсы полностью вырезаются компилятором и не создают дополнительного кода при выполнении.
*   Класс, реализующий интерфейс через ключевое слово `implements`, обязан полностью реализовать его сигнатуру.

#### Абстрактный класс (`abstract class`)
Абстрактный класс — это гибридная конструкция, которая задает общий контракт для наследников, но при этом может содержать конкретную реализацию полей и методов.
*   Нельзя создать экземпляр абстрактного класса напрямую (через оператор `new`).
*   Содержит абстрактные члены (с ключевым словом `abstract`), которые не имеют реализации и обязаны быть переопределены в классах-наследниках.

---

### Встроенные утилиты типизации (Utility Types)
TypeScript предоставляет набор обобщенных типов-трансформеров для манипуляции интерфейсами без дублирования кода:

*   **`Partial<T>`**: Преобразует все свойства типа `T` в необязательные (добавляет символ `?`).
*   **`Required<T>`**: Делает все свойства типа `T` строго обязательными (удаляет символ `?`).
*   **`Readonly<T>`**: Делает все свойства типа `T` доступными только для чтения.
*   **`Pick<T, K>`**: Создает новый тип, выбирая из исходного типа `T` только указанное подмножество ключей `K`.
*   **`Omit<T, K>`**: Противоположность `Pick`. Создает тип, исключая из исходного типа `T` ключи `K`.
*   **`Record<K, T>`**: Позволяет быстро сконструировать объектный тип (карту/словарь), где ключами являются элементы из `K`, а значениями — тип `T`.

## 2. Ментальная модель

### Сравнение структуры интерфейса, абстрактного и конкретного классов

```text
┌─────────────────────────────────┐
│           ИНТЕРФЕЙС             │ ◄─── Только контракт (без кода)
│  - getName(): string            │      [ Стирается в JS ]
└────────────────┬────────────────┘
                 ▲ (implements)
                 │
┌────────────────┴────────────────┐
│       АБСТРАКТНЫЙ КЛАСС         │ ◄─── Контракт + базовый код
│  - abstract getName(): string   │      [ Нельзя создать экземпляр ]
│  - getRole(): string { ... }    │
└────────────────┬────────────────┘
                 ▲ (extends)
                 │
┌────────────────┴────────────────┐
│        КОНКРЕТНЫЙ КЛАСС         │ ◄─── Полная реализация
│  - getName(): string { ... }    │      [ Можно вызвать new ]
│  - getRole(): string { ... }    │
└─────────────────────────────────┘
```

### Трансформация типов с помощью утилит
Утилиты типизации работают как конвейер преобразования схем данных:

```text
       Исходный тип (User)
      ┌────────────────────┐
      │ id: string         │
      │ name: string       │
      │ email: string      │
      └─────────┬──────────┘
                │
    ┌───────────┴───────────┐
    ▼ (Omit<User, 'id'>)    ▼ (Partial<User>)
┌───────────────┐       ┌────────────────┐
│ name: string  │       │ id?: string    │
│ email: string │       │ name?: string  │
└───────────────┘       │ email?: string │
                        └────────────────┘
```

## 3. Практические примеры кода

### Плохой пример: Избыточность кода, отсутствие readonly и ручное дублирование типов

```typescript
// Дублирование сущностей с разной степенью обязательности полей
interface UserEntity {
  id: string;
  username: string;
  email: string;
  createdAt: Date;
}

// Дублирование полей вручную для сценария обновления профиля
interface UpdateUserDto {
  username?: string;
  email?: string;
}

// Отсутствие инкапсуляции. Поля открыты для неконтролируемой внешней мутации
class UserServiceLegacy {
  public database: any; // un-typed

  constructor(db: any) {
    this.database = db;
  }

  public save(user: UserEntity) {
    this.database.push(user);
  }
}
```

### Хороший пример: Использование абстрактных классов, инкапсуляции и Utility Types

```typescript
interface UserEntity {
  id: string;
  username: string;
  email: string;
  createdAt: Date;
}

// Избегаем дублирования кода. Создаем типы на лету с помощью утилит
type UserUpdateDto = Partial<Omit<UserEntity, 'id' | 'createdAt'>>; // Только username? и email?
type UserSummary = Pick<UserEntity, 'id' | 'username'>; // Только id и username

// Абстрактный репозиторий, фиксирующий контракт для наследников
abstract class BaseRepository<T> {
  // Поле доступно только внутри этого класса и его наследников
  protected readonly dbConnection: string;

  constructor(connectionString: string) {
    this.dbConnection = connectionString;
  }

  // Абстрактный метод без реализации
  abstract save(entity: T): void;
}

// Реализация конкретного класса на основе абстрактного базового класса
class UserRepository extends BaseRepository<UserEntity> {
  // Record утилита для быстрого типизирования in-memory хранилища
  private readonly storage: Record<string, UserEntity> = {};

  constructor(connectionString: string) {
    super(connectionString);
  }

  // Обязательная реализация абстрактного метода родителя
  public save(user: UserEntity): void {
    this.storage[user.id] = user;
    console.log(`User saved to DB: ${this.dbConnection}`);
  }

  public update(id: string, updates: UserUpdateDto): void {
    const existing = this.storage[id];
    if (existing) {
      // Безопасное обновление полей
      this.storage[id] = { ...existing, ...updates };
    }
  }
}
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между `interface` и `type` в TypeScript? Когда что использовать?
**Ответ:** 
Несмотря на то что `type` (Type Alias) и `interface` во многих случаях взаимозаменяемы, между ними есть ключевые архитектурные различия:
1.  **Декларативное слияние (Declaration Merging)**: Интерфейсы с одинаковыми именами автоматически сливаются в один интерфейс. Псевдонимы типов (`type`) дублировать в рамках одной области видимости запрещено.
2.  **Расширяемость**: Интерфейсы расширяются через ключевое слово `extends`, типы — через пересечение типов (`&`).
3.  **Специфика**: Псевдонимы типов могут описывать примитивы, кортежи (tuples) и объединения (`|`), в то время как интерфейсы всегда описывают исключительно структуры объектов или функции.
**Рекомендация:** Используйте `interface` для описания публичных API сервисов, моделей данных и компонентов (для возможности последующего расширения и слияния). Используйте `type` для создания сложных типов, объединений (unions), кортежей или при утилитарных трансформациях типов.

### Вопрос 2: Как компилируются и ведут себя модификаторы доступа TypeScript (`private` / `protected`) во время выполнения кода в браузере? Как реализовать истинную инкапсуляцию?
**Ответ:** 
Модификаторы доступа `private`, `public` и `protected` существуют исключительно в экосистеме компилятора TypeScript. На этапе компиляции они стираются, и сгенерированный JS-код содержит обычные незащищенные свойства классов. Это значит, что во время выполнения (runtime) любой внешний скрипт может прочитать или изменить значение свойства, помеченного в TS как `private`.
Для реализации гарантированной runtime-изоляции (инкапсуляции) в современном стандарте ECMAScript (и TypeScript) используются приватные поля с префиксом `#` (например, `#storage`). Свойства с `#` компилируются со встроенной функцией изоляции движка JS, делая прямой доступ к ним извне невозможным на уровне спецификации JavaScript.

### Вопрос 3: Как утилита `Pick<T, K>` устроена под капотом? Напишите ее сигнатуру и объясните механизмы `keyof` и `in`.
**Ответ:** 
Встроенная утилита `Pick` базируется на механизме сопоставления типов (Mapped Types). Ее сигнатура выглядит следующим образом:
```typescript
type Pick<T, K extends keyof T> = {
    [P in K]: T[P];
};
```
*   `K extends keyof T` — накладывает ограничение на параметр `K`. Ключи, передаваемые в качестве `K`, обязаны быть подмножеством всех строковых литералов-ключей типа `T` (полученных через оператор `keyof T`).
*   `[P in K]` — итератор (аналог `for...in`), который последовательно обходит каждый строковый литерал `P` из переданного списка ключей `K`.
*   `T[P]` — обращение к типу свойства по индексу (Lookup Type), считывающее оригинальный тип свойства `P` из исходного типа `T`.

### Вопрос 4: Можно ли внутри `interface` объявить приватные (`private`) свойства? Обоснуйте ответ.
**Ответ:** 
Нет, внутри `interface` объявление приватных (`private`) или защищенных (`protected`) свойств невозможно и вызовет ошибку компиляции. 
Интерфейс по определению является контрактом, описывающим **публичный интерфейс** объекта (API), предназначенный для внешнего взаимодействия с ним. Описание внутренних деталей реализации и инкапсулированных данных противоречит фундаментальному назначению интерфейса. Все свойства и методы интерфейса неявно являются исключительно публичными (`public`). Для описания приватных контрактов следует использовать абстрактные классы.
