Маршрутизатор Angular (`@angular/router`) превращает приложение в полноценное одностраничное веб-приложение (SPA). Он управляет жизненным циклом компонентов на основе состояния адресной строки браузера, координирует асинхронную загрузку модулей и компонентов по требованию (Lazy Loading) и разграничивает права доступа к ресурсам с помощью механизмов контроля переходов (Guards).

---

## 1. Глубокое понимание концепции (Core Concept)

### Анатомия маршрутизации в Angular

Маршрутизация на стороне клиента работает по принципу сопоставления пути в URL с конфигурационным массивом объектов `Route`. Ключевые строительные блоки маршрутизатора:

1.  **Декларативное сопоставление (Routes):** Массив объектов конфигурации. Каждый объект связывает строковый шаблон пути (`path`) с конкретным компонентом (`component`) или асинхронной стратегией (`loadComponent`, `loadChildren`).
2.  **Точка монтирования (`RouterOutlet`):** Директива-слот `<router-outlet>`, куда Angular динамически встраивает экземпляр компонента, соответствующий текущему активному маршруту.
3.  **Декларативная навигация (`RouterLink`):** Директива, перехватывающая стандартное поведение браузера при переходе по ссылке `<a>`. Вместо перезагрузки страницы она передает управление маршрутизатору, сохраняя состояние приложения.
4.  **Программная навигация (`Router`):** Сервис, предоставляющий API для управления переходами из кода компонентов или сервисов через методы `navigate()` и `navigateByUrl()`.

#### Стратегии сопоставления (`pathMatch`)
При поиске соответствия Angular Router сканирует массив путей сверху вниз. Свойство `pathMatch` определяет глубину проверки соответствия URL шаблону пути:
*   `prefix` (по умолчанию): Маршрут подходит, если URL-адрес *начинается* с указанного пути. Например, путь `path: 'user'` совпадет с `/user`, `/user/1`, `/user/settings`.
*   `full`: Требует *строгого, полного совпадения* URL с шаблоном. Это критично для путей по умолчанию `path: ''` при перенаправлении (`redirectTo`), иначе префиксное совпадение приведет к бесконечному циклу редиректов.

#### Параметризованные URL и Вложенность
Параметры маршрута объявляются через двоеточие: `path: 'user/:id'`. Вложенные интерфейсы организуются через свойство `children`. Для отображения дочерних маршрутов родительский компонент обязательно должен содержать свой собственный независимый `<router-outlet>`.

---

### Ленивая загрузка (Lazy Loading)

Ленивая загрузка разделяет монолитный бандл приложения на изолированные асинхронные фрагменты (chunks), которые загружаются только тогда, когда пользователь непосредственно переходит по соответствующему адресу.

*   `loadComponent`: Используется для динамического импорта одного standalone-компонента. Принимает стрелочную функцию, возвращающую Promise динамического импорта.
*   `loadChildren`: Используется для ленивой загрузки группы дочерних маршрутов, описанных в отдельном файле конфигурации.

На уровне компилятора Angular CLI преобразует функции динамического импорта `() => import('./...')` в отдельные точки входа (JS-файлы), которые загружаются по сети методом JSONP/Fetch при активации маршрута.

---

### Механизмы контроля доступа (Guards)

Защитники (Guards) — это механизмы, определяющие возможность перехода на маршрут или ухода с него. В современном Angular все гарды представляют собой **функциональные выражения** (Functional Guards), заменяющие старые громоздкие классы. Они могут использовать механизм внедрения зависимостей через функцию `inject()`.

#### Типы Guards и порядок их выполнения:
1.  **`CanMatch`:** Проверяет, может ли маршрут вообще быть выбран для сопоставления. Если возвращает `false`, роутер игнорирует этот маршрут и ищет совпадение в массиве дальше. **Не загружает JS-бандл ленивого компонента в браузер в случае отказа.**
2.  **`CanActivate`:** Решает, разрешен ли доступ к запрашиваемому маршруту (например, проверка авторизации). Запускается *после* успешного сопоставления.
3.  **`CanActivateChild`:** Аналог `CanActivate`, проверяющий права доступа ко всем дочерним маршрутам текущей ветки.
4.  **`CanDeactivate`:** Решает, разрешено ли пользователю покинуть текущий маршрут (например, предупреждение о потере несохраненных изменений в форме).

## 2. Ментальная модель

Процесс обработки перехода по URL `https://your-app.com/admin/settings`:

```text
               Изменение URL в адресной строке (/admin/settings)
                                      │
                                      ▼
                       [Поиск маршрута в конфигурации]
                                      │
                                      ▼
             1. Проверка CanMatch (разрешено ли сопоставить путь?)
                    ├── Нет ──> [Поиск альтернативного пути]
                    └── Да
                                      │
                                      ▼
               2. Проверка CanDeactivate (разрешено уйти с текущего?)
                    ├── Нет ──> [Навигация заблокирована]
                    └── Да
                                      │
                                      ▼
                3. Проверка CanActivate (разрешено войти на новый?)
                    ├── Нет ──> [Навигация заблокирована / Редирект]
                    └── Да
                                      │
                                      ▼
             3. Загрузка чанка (если маршрут ленивый через loadComponent)
                                      │
                                      ▼
       5. Инстанцирование компонента -> Вставка в <router-outlet>
```

## 3. Практические примеры кода

### Плохой пример: Устаревшие классы-гарды и монолитный импорт всех компонентов

Импорт всех компонентов напрямую в корневую конфигурацию раздувает начальный бандл приложения (`main.js`). Использование устаревших классов-гардов усложняет чтение кода и увеличивает его объем.

```typescript
// app.routes.ts
import { Injectable } from '@angular/core';
import { Routes, CanActivate, Router } from '@angular/router';
import { Observable, of } from 'rxjs';

// Плохо: Все компоненты загружаются немедленно (Eager Loading), увеличивая размер начального бандла
import { HomeComponent } from './home/home.component';
import { AdminComponent } from './admin/admin.component';
import { AuthService } from './auth.service';

@Injectable({ providedIn: 'root' })
export class LegacyAuthGuard implements CanActivate {
  constructor(private auth: AuthService, private router: Router) {}

  // Плохо: Громоздкий классовый синтаксис, требующий бойлерплейта декораторов
  canActivate(): Observable<boolean> {
    if (this.auth.isLoggedIn()) {
      return of(true);
    }
    this.router.navigate(['/login']);
    return of(false);
  }
}

export const routes: Routes = [
  { path: '', component: HomeComponent },
  // Плохо: Пустой путь для редиректа без pathMatch: 'full' приведет к бесконечному рефакторингу URL
  { path: 'home', redirectTo: '' }, 
  { 
    path: 'admin', 
    component: AdminComponent, 
    canActivate: [LegacyAuthGuard] 
  }
];
```

### Хороший пример: Современная ленивая конфигурация с функциональными гардами

Использование `loadComponent` для деления кода на чанки, современных функциональных гардов через `inject` и активация привязки параметров маршрута к свойствам компонента.

```typescript
// app.routes.ts
import { Routes, Router } from '@angular/router';
import { inject } from '@angular/core';
import { AuthService } from './auth.service';

// Хорошо: Легковесный функциональный гард аутентификации
export const authGuard = () => {
  const authService = inject(AuthService);
  const router = inject(Router);

  return authService.isLoggedIn() ? true : router.createUrlTree(['/login']);
};

export const routes: Routes = [
  {
    path: '',
    pathMatch: 'full', // Хорошо: Строгое соответствие для корневого редиректа
    redirectTo: 'home'
  },
  {
    path: 'home',
    // Хорошо: Компонент будет загружен асинхронно только при переходе на /home
    loadComponent: () => import('./home/home.component').then(m => m.HomeComponent)
  },
  {
    path: 'admin',
    loadComponent: () => import('./admin/admin.component').then(m => m.AdminComponent),
    canActivate: [authGuard] // Хорошо: Использование чистого функционального гарда
  },
  {
    path: 'user/:id',
    loadComponent: () => import('./user/user.component').then(m => m.UserComponent)
  },
  {
    path: '**',
    loadComponent: () => import('./not-found/not-found.component').then(m => m.NotFoundComponent)
  }
];
```

```typescript
// app.config.ts
import { ApplicationConfig } from '@angular/core';
import { provideRouter, withComponentInputBinding } from '@angular/router';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    provideRouter(
      routes,
      // Хорошо: Автоматически связывает параметры маршрута (:id) и queryParams с @Input/Signal-вводами компонента
      withComponentInputBinding() 
    )
  ]
};
```

```typescript
// user.component.ts
import { Component, input } from '@angular/core';

@Component({
  selector: 'app-user',
  standalone: true,
  template: `<h1>ID Пользователя из URL: {{ id() }}</h1>`
})
export class UserComponent {
  // Хорошо: Благодаря скомпонованному withComponentInputBinding() значение id биндится автоматически в Signal
  id = input.required<string>(); 
}
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между `CanActivate` и `CanMatch`? Какое преимущество дает `CanMatch` при ленивой загрузке ресурсов?
**Ответ:** 
*   **`CanActivate`** срабатывает *после* того, как маршрут был успешно сопоставлен с URL. В случае ленивой загрузки (`loadComponent`/`loadChildren`), браузер **сначала скачивает JS-бандл** компонента по сети, и только потом запускает проверку `CanActivate`. Если гард вернет `false`, навигация заблокируется, но исходный код страницы уже будет доставлен на клиент, что небезопасно и неоптимально.
*   **`CanMatch`** срабатывает на самом раннем этапе сопоставления URL с конфигурацией. Проверка происходит *до* загрузки асинхронного ресурса. Если `CanMatch` возвращает `false`, роутер не загружает JS-бандл, прекращает рассматривать данный маршрут и пытается сопоставить URL со следующими правилами в конфигурации (что позволяет реализовать, например, выдачу разных компонентов по одному URL на базе A/B тестирования или ролей).

### Вопрос 2: Объясните разницу между поведением `pathMatch: 'full'` и `pathMatch: 'prefix'`. Почему для пути `path: ''` необходим `full`?
**Ответ:** 
*   `pathMatch: 'prefix'` сообщает маршрутизатору, что сопоставление считается успешным, если текущий URL начинается с указанной в `path` строки.
*   `pathMatch: 'full'` требует полного и точного совпадения остатка URL с шаблоном.
Если мы настроим маршрут по умолчанию `{ path: '', redirectTo: 'home' }` со стратегией по умолчанию `prefix`, Angular попадет в бесконечный цикл. Так как пустая строка `''` является префиксом для любого URL-адреса, роутер будет пытаться применить редирект для каждого перехода (даже когда URL уже равен `/home`), приводя к ошибке перенаправления. Использование `pathMatch: 'full'` гарантирует, что редирект сработает только тогда, когда в адресной строке нет дополнительных сегментов пути.

### Вопрос 3: Как работает механизм `CanDeactivate`? Опишите сценарий предотвращения случайной потери данных при заполнении форм.
**Ответ:** 
`CanDeactivate` запускается при попытке пользователя покинуть текущую страницу. Данный гард имеет доступ к инспектированию состояния текущего компонента. 
Функциональный `CanDeactivate` принимает экземпляр активного компонента в качестве первого аргумента. Внутри гарда проверяется флаг «загрязнения» формы (например, `component.form.dirty` или наличие несохраненного состояния). Если изменения есть, гард открывает модальное диалоговое окно (нативное `confirm` или кастомное через Observable-сервис). Возвращая `true` или `false` (или поток `Observable<boolean>`), гард разрешает навигацию дальше или блокирует смену URL, сохраняя данные пользователя.

### Вопрос 4: Как с помощью `withComponentInputBinding()` передавать статические и динамические данные в свойства компонента без внедрения `ActivatedRoute`?
**Ответ:** 
При вызове `provideRouter(routes, withComponentInputBinding())` Angular Router автоматически связывает параметры маршрута со свойствами компонента. Он сопоставляет имена параметров из трех источников:
1.  Параметры пути (например, `:id` из `user/:id`).
2.  Параметры запроса (Query parameters, например, `?tab=details`).
3.  Статические данные, переданные в свойстве конфигурации `data: { title: 'Settings' }`.

Если в целевом компоненте объявлены свойства `@Input() id` или Signal `id = input<string>()`, маршрутизатор автоматически внедрит туда соответствующие значения при инициализации и обновлении маршрута, избавляя от необходимости внедрять сервис `ActivatedRoute` и вручную подписываться на объект `params` или `queryParams`.
