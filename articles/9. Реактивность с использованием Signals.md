Сигналы (Signals) — это легковесные реактивные обертки над изменяемыми значениями, которые коренным образом меняют способ управления состоянием и отслеживания изменений в Angular. Появившись в версии 16 и став стабильными в версии 17, Сигналы позволяют выстраивать синхронный декларативный граф зависимостей. Они точечно уведомляют фреймворк об изменениях конкретных значений, что делает возможным переход к высокопроизводительному рендерингу без Zone.js (Zoneless).

---

## 1. Глубокое понимание концепции (Core Concept)

### Природа Сигналов и Реактивный контекст
Сигнал представляет собой значение, чтение которого регистрируется платформой Angular. Код, который считывает значение сигнала, называется потребителем (consumer), а сам считываемый сигнал — производителем (producer). 

Когда сигнал считывается внутри специального исполняемого окружения, Angular автоматически переходит в **реактивный контекст** и связывает потребителя с производителем. 

Реактивный контекст создается в следующих случаях:
*   При рендеринге HTML-шаблона компонента.
*   Внутри вычисляемых сигналов при вызове функции `computed`.
*   При синхронизации значений в `linkedSignal`.
*   Внутри колбэков побочных эффектов при вызове функций `effect` или `afterRenderEffect`.

Если значение сигнала-производителя меняется, Angular автоматически помечает всех его зарегистрированных потребителей как грязные для последующего обновления.

---

### Разновидности Сигналов

#### 1. Изменяемые сигналы (`WritableSignal`)
Создаются с помощью базовой функции `signal` с указанием начального значения. Предоставляют API для прямого обновления данных:
*   Метод `.set()` заменяет значение сигнала на новое.
*   Метод `.update()` вычисляет новое значение на основе предыдущего с помощью переданной функции.

С помощью метода `.asReadonly()` изменяемый сигнал можно привести к типу `Signal` только для чтения, скрыв методы записи от внешних потребителей.

#### 2. Вычисляемые сигналы (`computed`)
Создаются декларативно на основе других сигналов. Они всегда доступны только для чтения и вычисляются лениво (ленивые вычисления):
*   Функция вычисления не выполняется до тех пор, пока значение `computed` не будет считано первый раз.
*   Результат вычисления кешируется. Повторные чтения возвращают кешированное значение без пересчета.
*   Если исходные сигналы-производители меняются, кеш инвалидируется, и новое значение вычисляется при следующем чтении.

#### 3. Связанные сигналы (`linkedSignal`)
Концепция, ставшая стабильной в Angular 20, решает проблему синхронизации зависимого состояния. `linkedSignal` — это изменяемый сигнал, значение которого сбрасывается к дефолтному при изменении внешнего источника (сигнала-родителя), но при этом может быть изменено вручную через методы записи в любой момент без ожидания изменения источника.

---

### Побочные эффекты (`effect`) и функция `untracked`
Функция `effect` регистрирует операцию, которая автоматически перезапускается каждый раз, когда меняется один или несколько считанных внутри нее сигналов.

*   Эффекты всегда запускаются асинхронно в процессе обнаружения изменений и выполняются как минимум один раз при инициализации.
*   Эффекты должны создаваться внутри контекста внедрения зависимостей (например, в конструкторе) или явно получать объект `Injector` в параметрах.
*   Использование эффектов для перезаписи данных из одного сигнала в другой считается плохой практикой (для этого нужно использовать `computed` или `linkedSignal`). Эффекты созданы для интеграции с внешними API: логирование, синхронизация с локальным хранилищем (`localStorage`) или работа с нативным DOM (отрисовка на Canvas, интеграция чартов).

#### Функция `untracked`
Иногда внутри эффекта или вычисляемого сигнала требуется прочитать значение какого-либо сигнала, не создавая при этом реактивной зависимости от него. Для этого чтение оборачивается в функцию `untracked()`.

---

### Сравнение RxJS и Signals. Интероперабельность
Сигналы и потоки RxJS дополняют друг друга, но решают разные задачи:
*   **Signals** работают строго синхронно. Они идеальны для хранения локального UI-состояния компонентов, так как обеспечивают простой вызов в шаблонах без подписок.
*   **RxJS** ориентирован на асинхронность и сложные декларативные трансформации (работа с HTTP, веб-сокетами, таймерами и событиями отмены).

Для их совместной интеграции библиотека `@angular/core/rxjs-interop` предоставляет утилиты:
*   **`toSignal`**: преобразует асинхронный поток Observable в сигнал только для чтения.
*   **`toObservable`**: преобразует сигнал в поток Observable, который генерирует новое событие при каждом изменении значения сигнала.

## 2. Ментальная модель

### Синхронный граф зависимостей Сигналов

```text
    [ WritableSignal: price ] ───┐
                                 ▼
    [ WritableSignal: tax ] ───► [ computed: totalPrice ] ───► [ Шаблон DOM ]
                                 ▲
    [ WritableSignal: count ] ───┘
```
Если меняется `price`, `tax` или `count`, Angular точно знает, что кеш `totalPrice` недействителен, и точечно перерисовывает только тот участок шаблона, который отображает общее значение.

## 3. Практические примеры кода

### Плохой пример: Попытка ручной синхронизации сигналов через эффекты

В данном примере разработчик копирует данные из одного сигнала в другой с помощью эффекта. Это создает избыточные циклы рендеринга и нарушает реактивную цепочку. Также пример демонстрирует нецелевое использование эффектов.

```typescript
import { Component, signal, effect } from '@angular/core';

@Component({
  selector: 'app-bad-signals',
  standalone: true,
  template: `
    <div>Сумма: {{ total() }}</div>
  `
})
export class BadSignalsComponent {
  price = signal(100);
  quantity = signal(2);
  total = signal(200);

  constructor() {
    // Ошибка: Ручная синхронизация через эффект является антипаттерном.
    // Это создает лишние триггеры записи и усложняет отладку.
    effect(() => {
      const calculated = this.price() * this.quantity();
      this.total.set(calculated);
    });
  }
}
```

### Хороший пример: Использование computed, linkedSignal и интеропа с RxJS

```typescript
// src/app/components/product-calculator/product-calculator.component.ts
import { Component, signal, computed, linkedSignal, inject } from '@angular/core';
import { toSignal } from '@angular/core/rxjs-interop';
import { HttpClient } from '@angular/common/http';
import { map } from 'rxjs/operators';

interface ProductDiscount {
  rate: number;
}

@Component({
  selector: 'app-product-calculator',
  standalone: true,
  template: `
    <div class="calculator">
      <h3>Товар: {{ productName() }}</h3>
      <p>Базовая цена: {{ basePrice() }} руб.</p>
      
      <!--linkedSignal автоматически обновляет скидку при смене товара, но разрешает ручной ввод -->
      <p>Текущая скидка: {{ discount() }}%</p>
      <button (click)="setPromoDiscount()">Применить супер-промо (50%)</button>
      <button (click)="changeProduct()">Сменить товар</button>

      <h4>Итоговая стоимость: {{ finalPrice() }} руб.</h4>
    </div>
  `
})
export class ProductCalculatorComponent {
  private readonly http = inject(HttpClient);

  // Динамическое получение скидки из API и преобразование потока RxJS в Сигнал
  private readonly apiDiscount$ = this.http.get<ProductDiscount>('/api/discount').pipe(
    map(res => res.rate)
  );
  readonly externalDiscount = toSignal(this.apiDiscount$, { initialValue: 5 });

  // Локальные изменяемые сигналы
  readonly productName = signal('Ноутбук');
  readonly basePrice = signal(80000);

  // linkedSignal решает проблему синхронизации. 
  // При изменении родительского товара скидка автоматически сбрасывается к значению из API.
  readonly discount = linkedSignal({
    source: this.productName,
    computation: () => this.externalDiscount()
  });

  // computed декларативно рассчитывает итоговую цену без побочных эффектов
  readonly finalPrice = computed(() => {
    const priceValue = this.basePrice();
    const discountMultiplier = 1.0 - this.discount() / 100;
    return priceValue * discountMultiplier;
  });

  setPromoDiscount(): void {
    // linkedSignal позволяет перезаписать значение вручную извне
    this.discount.set(50);
  }

  changeProduct(): void {
    this.productName.set('Смартфон');
    this.basePrice.set(30000);
    // При этом вызове discount автоматически сбросится к актуальному значению externalDiscount()
  }
}
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между вычисляемым сигналом `computed` и эффектом `effect`?
**Ответ:** 
*   `computed` предназначен для вычисления новых производных состояний на основе других сигналов. Он не должен совершать никаких побочных действий (изменение DOM, отправка запросов). Вычисления в `computed` являются ленивыми, кешируются и происходят только тогда, когда результат кто-то считывает. Он всегда возвращает сигнал только для чтения.
*   `effect` предназначен исключительно для выполнения побочных действий вне реактивного контекста Angular (логирование, синхронизация с хранилищем, отрисовка графики). Он выполняется асинхронно, запускается как минимум один раз и всегда реагирует на изменения зависимостей, не возвращая никакого значения.

### Вопрос 2: Какую архитектурную проблему решает концепция `linkedSignal`, появившаяся в Angular 20?
**Ответ:** 
`linkedSignal` решает проблему синхронизации зависимого состояния (state synchronization). Ранее, если требовалось сбросить значение изменяемого сигнала при изменении другого сигнала, приходилось писать императивный код внутри `effect` или создавать сложные обертки. Это нарушало однонаправленный поток данных и усложняло отладку из-за риска зацикливания реактивности. `linkedSignal` объединяет в себе свойства реактивного сброса по изменению источника и возможность ручной императивной перезаписи, предоставляя единый безопасный источник правды.

### Вопрос 3: Как работает ленивое вычисление (Lazy Evaluation) и кеширование в `computed` сигналах?
**Ответ:** 
Когда вы объявляете `computed`, Angular не запускает функцию вычисления мгновенно. Он лишь сохраняет переданный колбэк. Само вычисление происходит только тогда, когда кто-то впервые считывает значение вычисляемого сигнала (например, при рендеринге шаблона). 
После этого Angular кеширует полученный результат. При последующих чтениях возвращается сохраненное значение без повторного выполнения функции. Перерасчет произойдет только в том случае, если исходные сигналы, от которых зависит `computed`, изменили свои значения, и то — строго при следующем обращении к этому вычисляемому сигналу.

### Вопрос 4: Зачем нужна функция `untracked` и в каких ситуациях ее применение необходимо?
**Ответ:** 
Функция `untracked` используется в тех случаях, когда внутри реактивной функции (в `effect`, `computed` или `linkedSignal`) требуется прочитать значение сигнала, не создавая при этом нежелательную зависимость от него. 
**Пример использования**: Эффект должен отправлять лог аналитики на сервер при изменении значения корзины товаров `cart()`. В логе мы также хотим передать имя текущего пользователя `user()`. Если мы запишем это напрямую, эффект будет запускаться и при смене имени пользователя. Обернув чтение `user()` в `untracked(() => this.user())`, мы гарантируем, что эффект будет срабатывать строго при изменении корзины, но при этом сможет безопасно считать актуальное имя пользователя.
