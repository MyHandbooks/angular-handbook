Angular — это компонентно-ориентированная платформа и фреймворк от компании Google, предназначенный для проектирования, сборки и сопровождения масштабируемых одностраничных веб-приложений (SPA). В отличие от библиотек рендеринга, Angular предоставляет полноценную экосистему из коробки, стандартизируя подходы к маршрутизации, управлению формами, сетевому взаимодействию и внедрению зависимостей.

---

## 1. Глубокое понимание концепции (Core Concept)

### Окружение выполнения и система управления зависимостями (Node.js, npm)
Современная разработка на Angular опирается на среду выполнения **Node.js** и менеджер пакетов **npm** (Node Package Manager). Сама платформа Angular выполняется исключительно в браузере клиента, однако Node.js критически необходим на этапе разработки для функционирования компиляторов, линтеров, сборщиков (Webpack, Vite/Esbuild) и тестовых сред.

Важнейшим элементом управления проектом является манифест **package.json** и файл блокировки версий **package-lock.json**:
*   **package.json**: Описывает метаданные проекта, скрипты автоматизации (`npm start`, `npm run build`), а также разделяет зависимости на runtime (`dependencies`) и dev-time (`devDependencies`).
*   **Правила фиксации версий (SemVer)**:
    *   `^1.4.7` (Caret) — разрешает обновление минорных версий и патчей (установит до `< 2.0.0`). Подходит при высоком доверии к обратной совместимости библиотек.
    *   `~1.4.7` (Tilde) — разрешает только патч-обновления (установит до `< 1.5.0`). Используется для минимизации рисков нарушения работоспособности кода.
    *   `1.4.7` (Exact) — строго фиксированная версия. Исключает неконтролируемые обновления.
*   **package-lock.json**: Фиксирует абсолютно точное дерево зависимостей, включая транзитивные (зависимости зависимостей), гарантируя идентичность сборки на машинах всех участников команды разработки и на CI/CD-серверах.

### Анатомия Angular-приложения и переход к Standalone-парадигме
Исторически архитектура Angular опиралась на модули (`NgModule`), которые группировали связанные компоненты, директивы, пайпы и провайдеры. Начиная с Angular 14/15 и окончательно закрепившись в качестве стандарта в последних версиях (включая Angular 19), фреймворк перешел на **Standalone-компоненты**.

Принципиальные отличия двух подходов:

| Критерий                      | Подход с использованием NgModule                     | Standalone-подход                                              |
| :---------------------------- | :--------------------------------------------------- | :------------------------------------------------------------- |
| **Единица декомпозиции**      | Модуль (класс с декоратором `@NgModule`)             | Компонент (класс с декоратором `@Component`)                   |
| **Декларация зависимостей**   | Описывается централизованно в модуле                 | Описывается непосредственно в самом компоненте через `imports` |
| **Сложность tree-shaking**    | Затруднена из-за связанности элементов внутри модуля | Максимально упрощена благодаря прямой импортации               |
| **Инициализация (Bootstrap)** | Через `bootstrapModule(AppModule)`                   | Через `bootstrapApplication(AppComponent, appConfig)`          |

В современной standalone-архитектуре точка входа `src/main.ts` напрямую инициализирует корневой компонент с помощью функции `bootstrapApplication`, передавая объект конфигурации `ApplicationConfig` (содержащий провайдеры для роутинга, HTTP-клиента и других глобальных систем).

## 2. Ментальная модель

Ниже приведена схема процесса сборки и инициализации (bootstrapping) современного standalone-приложения Angular:

```text
[ БРАУЗЕР ]
    │
    ▼ (Загрузка index.html)
┌───────────────────────────────────────┐
│ index.html                            │
│  └─ <app-root></app-root>             │
└───────┬───────────────────────────────┘
        │
        ▼ (Выполнение скомпилированного bundle)
┌───────────────────────────────────────┐
│ main.ts                               │
│  └─ bootstrapApplication(App, config) │
└───────┬───────────────────────┬───────┘
        │                       │
        │ (Конфигурация)        │ (Отрисовка в DOM)
        ▼                       ▼
┌───────────────────┐   ┌─────────────────────────────────────┐
│ app.config.ts     │   │ app.component.ts (Root Component)   │
│  ├─ provideRouter │   │  ├─ selector: 'app-root'            │
│  └─ ...providers  │   │  ├─ template / HTML-разметка        │
└───────────────────┘   │  └─ styles / CSS                    │
                        └─────────────────────────────────────┘
```

**Пояснение к модели**: 
Браузер считывает `index.html` и находит пустой тег-селектор `<app-root>`. При выполнении бандла в `main.ts` вызывается функция `bootstrapApplication`, которая считывает метаданные из `app.component.ts` и `app.config.ts`. Angular инициализирует DI-контейнер приложения, создает экземпляр `AppComponent`, рендерит его шаблон и монтирует результат на место тега `<app-root>`.

## 3. Практические примеры кода

### Плохой пример: Устаревший подход на основе NgModule (с избыточным бойлерплейтом)

В старых проектах для связывания компонентов требовалось создавать тяжеловесные модули. Это размывало логику зависимостей компонента и усложняло статический анализ кода сборщиками.

```typescript
// app.component.ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-root',
  template: '<h1>Привет, {{ title }}</h1>'
})
export class AppComponent {
  title = 'Legacy App';
}

// app.module.ts (Лишний уровень абстракции)
import { NgModule } from '@angular/core';
import { BrowserModule } from '@angular/platform-browser';
import { AppComponent } from './app.component';

@NgModule({
  declarations: [AppComponent], // Ручное декларирование компонента
  imports: [BrowserModule],
  providers: [],
  bootstrap: [AppComponent]
})
export class AppModule { }

// main.ts
import { platformBrowserDynamic } from '@angular/platform-browser-dynamic';
import { AppModule } from './app/app.module';

// Бойлерплейт-компиляция модуля в рантайме (JIT-подход по умолчанию для старых версий)
platformBrowserDynamic().bootstrapModule(AppModule)
  .catch(err => console.error(err));
```

### Хороший пример: Современный декларативный Standalone-подход

В современной архитектуре Angular компоненты полностью независимы, а инициализация выполняется функционально и лаконично.

```typescript
// src/app/app.config.ts
import { ApplicationConfig, provideZoneChangeDetection } from '@angular/core';
import { provideRouter } from '@angular/router';
import { routes } from './app.routes';

export const appConfig: ApplicationConfig = {
  providers: [
    // Оптимизация производительности: коалесценция событий в Zone.js
    provideZoneChangeDetection({ eventCoalescing: true }),
    provideRouter(routes)
  ]
};

// src/app/app.component.ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-root',
  standalone: true, // Флаг обязателен (в Angular 19+ активен по умолчанию)
  imports: [], // Любые внешние зависимости (директивы, другие компоненты) импортируются прямо сюда
  template: `
    <main class="container">
      <h1>Привет, {{ title }}</h1>
    </main>
  `,
  styles: [`
    .container {
      max-width: 1200px;
      margin: 0 auto;
      padding: 1rem;
    }
  `]
})
export class AppComponent {
  title = 'Modern Angular';
}

// src/main.ts
import { bootstrapApplication } from '@angular/platform-browser';
import { AppComponent } from './app/app.component';
import { appConfig } from './app/app.config';

// Быстрый запуск standalone-компонента с глобальной конфигурацией
bootstrapApplication(AppComponent, appConfig)
  .catch(err => console.error(err));
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между командами `npm install` и `npm ci`?
**Ответ:** 
*   `npm install` анализирует файл `package.json`, вычисляет допустимые версии зависимостей на основе префиксов (`^`, `~`) и при наличии возможности обновляет `package-lock.json` и директорию `node_modules`.
*   `npm ci` (Clean Install) строго следует структуре `package-lock.json`. Она полностью удаляет текущую папку `node_modules`, проверяет соответствие `package.json` и `package-lock.json` (выдает ошибку при несовпадении версий) и устанавливает точные зафиксированные версии без изменения конфигурационных файлов. Эта команда обязательна к использованию на сборочных серверах (CI) для обеспечения детерминированности сборок.

### Вопрос 2: Как работает механизм компиляции JIT и AOT в Angular, и какой из них используется при сборке продакшна?
**Ответ:** 
*   **JIT (Just-In-Time)** компиляция переводит HTML-шаблоны и декораторы Angular в исполняемый JavaScript-код непосредственно в браузере пользователя во время загрузки страницы. Это увеличивает размер бандла (так как компилятор Angular `@angular/compiler` должен быть доставлен клиенту) и замедляет первичную отрисовку. Используется преимущественно в режиме локальной разработки для быстрой сборки (Hot Module Replacement).
*   **AOT (Ahead-Of-Time)** компиляция выполняет перевод кода Angular на этапе сборки проекта (build time). В браузер доставляется чистый оптимизированный JavaScript-код, не требующий компилятора в бандле. Это повышает безопасность (шаблоны скомпилированы и защищены от инъекций), уменьшает размер передаваемых данных и значительно ускоряет первую отрисовку страницы (FCP). В продакшн-сборках используется исключительно AOT.

### Вопрос 3: Зачем нужен файл `angular.json` и какую роль в нем играют "бюджеты" (Budgets)?
**Ответ:** 
`angular.json` — это манифест конфигурации рабочей области Angular CLI. Он определяет структуру проектов в репозитории, пути к исходному коду, настройки компиляции, стилей, ассетов и различных сред (development, production).
Секция `budgets` внутри конфигурации сборки позволяет задавать предельные размеры для генерируемых файлов бандла (например, warning при 500 Кб, error при 1 Мб). Это предотвращает случайное добавление тяжелых сторонних библиотек и контролирует общий размер приложения на этапе CI/CD, блокируя деплой слишком "раздутых" сборок.

### Вопрос 4: Как именно переход к Standalone-компонентам упрощает процесс оптимизации (Tree Shaking) сборщиками приложений?
**Ответ:** 
В традиционной архитектуре с `NgModule` компилятор не мог однозначно определить на этапе статического анализа, какие именно компоненты или директивы модуля реально используются в шаблонах. Из-за этого сборщики были вынуждены включать в итоговый бандл весь импортированный модуль целиком. 
В standalone-компонентах все зависимости (компоненты, директивы, пайпы) импортируются явно через свойство `imports` декоратора `@Component`. Сборщик (например, Esbuild или Webpack) анализирует граф импортов конкретных файлов, что позволяет легко исключить неиспользуемый код (dead-code elimination/tree-shaking) на уровне отдельных компонентов, значительно снижая общий вес итоговых файлов.
