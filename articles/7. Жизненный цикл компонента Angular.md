Каждый компонент в Angular проходит через строго определенный жизненный цикл с момента своего создания до удаления из DOM-дерева. Фреймворк автоматически управляет этим процессом, предоставляя разработчику точки интеграции (хуки жизненного цикла) для выполнения инициализации данных, реагирования на внешние изменения, управления рендерингом и очистки ресурсов. Одновременно с этим Angular позволяет расширять шаблоны за счет гибкой проекции внешнего контента.

---

## 1. Глубокое понимание концепции (Core Concept)

### Последовательность выполнения хуков жизненного цикла (Lifecycle Hooks)
При создании и обновлении компонента Angular последовательно вызывает ряд методов класса. Для использования этих методов в TypeScript-классах рекомендуется явно импортировать и реализовывать соответствующие интерфейсы (название интерфейса совпадает с названием хука без префикса `ng`).

Хуки вызываются в строго определенном порядке:

1.  **`ngOnChanges` (интерфейс `OnChanges`)**: Вызывается при первичной установке и при каждом последующем изменении значений входных свойств (декораторов `@Input` или сигналов `input()`). Метод принимает объект `SimpleChanges`, содержащий текущие и предыдущие значения измененных свойств.
2.  **`ngOnInit` (интерфейс `OnInit`)**: Вызывается ровно один раз сразу после первого вызова `ngOnChanges`. Здесь инициализируются обычные свойства класса и выполняются первые запросы к серверам или сервисам. На этом этапе DOM-шаблон компонента еще не гарантирует доступность дочерних элементов.
3.  **`ngDoCheck` (интерфейс `DoCheck`)**: Запускается при каждой проверке изменений в приложении. Позволяет вручную отслеживать изменения, которые Angular не может обнаружить автоматически.
4.  **`ngAfterContentInit` (интерфейс `AfterContentInit`)**: Вызывается один раз после того, как Angular проецирует внешнее содержимое (контент, переданный между тегами компонента) в шаблон самого компонента.
5.  **`ngAfterContentChecked` (интерфейс `AfterContentChecked`)**: Вызывается после каждой проверки спроецированного контента.
6.  **`ngAfterViewInit` (интерфейс `AfterViewInit`)**: Вызывается один раз после полной инициализации представления компонента и всех его дочерних элементов. Начиная с этого момента можно безопасно работать с запросами шаблона, такими как `@ViewChild`.
7.  **`ngAfterViewChecked` (интерфейс `AfterViewChecked`)**: Вызывается после каждой проверки представления компонента и его дочерних элементов на наличие изменений.
8.  **`ngOnDestroy` (интерфейс `OnDestroy`)**: Вызывается непосредственно перед уничтожением компонента и его удалением из DOM-дерева. Здесь выполняются все очистки: отписка от асинхронных потоков, отмена таймеров и удаление глобальных слушателей событий.

### Проекция контента
Для создания гибких UI-компонентов (например, карточек, модальных окон или табов) Angular предоставляет механизм проекции контента с помощью встроенной директивы `<ng-content>`. Она служит заполнителем (placeholder) для разметки, которую пользователь передает между тегами вызова компонента.

#### Атрибут `select`
Если компонент должен распределять переданный извне контент по разным частям своего шаблона, используется мультипроекция с атрибутом `select`. В качестве значения атрибута указывается CSS-селектор элемента:
*   `<ng-content select="[header]"></ng-content>` — спроецирует только те элементы, которые имеют атрибут `header`.
*   `<ng-content></ng-content>` — без атрибута `select` спроецирует весь оставшийся контент, для которого не нашлось подходящих селекторов.

## 2. Ментальная модель

### Схема вызовов жизненного цикла компонента

```text
    [ Создание экземпляра класса (constructor) ]
                        │
                        ▼
         [ ngOnChanges (при наличии Inputs) ]
                        │
                        ▼
                  [ ngOnInit ]
                        │
                        ▼
                  [ ngDoCheck ]
                        │
            ┌───────────┴───────────┐
            ▼                       ▼
    [ ngAfterContentInit ]   [ ngAfterViewInit ]
            │                       │
            ▼                       ▼
  [ ngAfterContentChecked ] [ ngAfterViewChecked ]
            │                       │
            └───────────┬───────────┘
                        │ (При уничтожении компонента)
                        ▼
                  [ ngOnDestroy ]
```

## 3. Практические примеры кода

### Плохой пример: Опасная императивная логика в неподходящих хуках

В данном примере разработчик пытается работать с элементами представления до их полной инициализации, а также провоцирует бесконечный цикл проверок из-за неверного использования методов отслеживания.

```typescript
import { Component, OnInit, DoCheck } from '@angular/core';

@Component({
  selector: 'app-bad-lifecycle',
  standalone: true,
  template: `
    <div id="element">Контент</div>
  `
})
export class BadLifecycleComponent implements OnInit, DoCheck {
  
  ngOnInit(): void {
    // Ошибка: Элемент еще не отрендерен в DOM. displayElement будет равен null
    const displayElement = document.getElementById('element');
    if (displayElement) {
      displayElement.style.color = 'blue';
    }
  }

  ngDoCheck(): void {
    // Ошибка: DoCheck вызывается при каждой микрозадаче. 
    // Тяжелые операции здесь критически замедлят приложение.
    this.runHeavyValidation();
  }

  private runHeavyValidation() {
    console.log('Выполняем проверку при каждом событии в приложении');
  }
}
```

### Хороший пример: Безопасное проецирование и использование хуков жизненного цикла

В этом примере дочерние элементы безопасно настраиваются в предназначенных для этого хуках, а компонент разметки использует мультислотовую проекцию.

```typescript
// src/app/components/card/card.component.ts
import { Component } from '@angular/core';

@Component({
  selector: 'app-card',
  standalone: true,
  template: `
    <div class="card">
      <div class="card-header">
        <!-- Сюда попадет только элемент с атрибутом card-header -->
        <ng-content select="[card-header]"></ng-content>
      </header>
      
      <div class="card-body">
        <!-- Сюда попадет весь контент без специфичных атрибутов -->
        <ng-content></ng-content>
      </div>
    </div>
  `,
  styles: [`
    .card { border: 1px solid #e0e0e0; border-radius: 6px; }
    .card-header { padding: 0.5rem; border-bottom: 1px solid #e0e0e0; }
    .card-body { padding: 1rem; }
  `]
})
export class CardComponent {}
```

```typescript
// src/app/components/profile/profile.component.ts
import { Component, OnInit, AfterViewInit, OnDestroy, ViewChild, ElementRef, signal } from '@angular/core';
import { Subscription, interval } from 'rxjs';

@Component({
  selector: 'app-profile',
  standalone: true,
  template: `
    <div class="profile-card">
      <h2 #titleText>Профиль пользователя</h2>
      <p>Время сессии: {{ activeTime() }} сек.</p>
    </div>
  `
})
export class ProfileComponent implements OnInit, AfterViewInit, OnDestroy {
  // Безопасный запрос к элементу шаблона
  @ViewChild('titleText') titleRef!: ElementRef<HTMLHeadingElement>;

  readonly activeTime = signal(0);
  private timerSubscription?: Subscription;

  // Безопасная инициализация асинхронных операций
  ngOnInit(): void {
    this.timerSubscription = interval(1000).subscribe(val => {
      this.activeTime.set(val);
    });
  }

  // Безопасная работа с DOM-представлением после его полной отрисовки
  ngAfterViewInit(): void {
    this.titleRef.nativeElement.style.color = '#2563eb';
  }

  // Обязательная очистка ресурсов при уничтожении компонента
  ngOnDestroy(): void {
    if (this.timerSubscription) {
      this.timerSubscription.unsubscribe();
    }
  }
}
```

## 4. Вопросы для собеседования

### Вопрос 1: Почему `ngAfterContentInit` выполняется раньше, чем `ngAfterViewInit` в жизненном цикле компонента?
**Ответ:** 
Это связано со спецификой построения иерархии DOM-дерева в Angular:
*   **Спроецированный контент (Content)** объявляется и создается в шаблоне *родительского* компонента, поэтому он готов и скомпилирован еще до того, как дочерний компонент начнет собирать собственное представление. Angular сначала внедряет готовый спроецированный контент на место тегов `<ng-content>`, вызывая `ngAfterContentInit`.
*   **Собственное представление (View)** включает внутренний HTML-шаблон дочернего компонента и все его вложенные директивы. Angular может подтвердить полную готовность этой структуры только после того, как разберется с внешним спроецированным контентом, поэтому `ngAfterViewInit` вызывается позже.

### Вопрос 2: Как поведет себя хук `ngOnChanges`, если изменить свойство внутри переданного объекта (по ссылке), не меняя саму ссылку на объект?
**Ответ:** 
Хук `ngOnChanges` не выполнится. Для оптимизации производительности Angular использует поверхностное сравнение (Shallow Comparison) входных параметров. Он сверяет новые значения с предыдущими по ссылке в памяти. Поскольку изменение внутреннего свойства объекта не меняет адрес его ссылки в куче, Angular считает, что входной параметр не изменился, и пропускает вызов `ngOnChanges`. Чтобы отслеживать такие изменения, необходимо либо передавать новые иммутабельные копии объектов (созданные через деструктуризацию), либо использовать реактивные сигналы (Signals) в качестве входных свойств.

### Вопрос 3: К какому багу приводит попытка изменить состояние компонента (значение переменной шаблона) внутри хука `ngAfterViewInit`? Как этого избежать?
**Ответ:** 
Это приводит к ошибке `ExpressionChangedAfterItHasBeenCheckedError` в режиме разработки. Она возникает из-за того, что в цикле Change Detection Angular сначала вычисляет значения шаблона и рендерит их, а затем проводит повторную верификационную проверку. Если изменить переменную состояния внутри `ngAfterViewInit` (после того, как рендеринг уже завершился), Angular обнаружит несовпадение данных между рендером и моделью. Чтобы избежать этой ошибки, не следует менять состояние компонента внутри хуков отображения. Если это необходимо, изменение нужно обернуть в `setTimeout` (чтобы перенести его выполнение в следующий цикл выполнения макрозадач) или запустить ручной цикл проверок изменений с помощью метода `detectChanges()` сервиса `ChangeDetectorRef`.

### Вопрос 4: Как работает мультислотовая проекция контента и что произойдет, если несколько тегов `<ng-content>` совпадут с одним и тем же DOM-элементом?
**Ответ:** 
Мультислотовая проекция использует атрибут `select` для сопоставления элементов по CSS-селекторам (атрибутам, классам или тегам). Элементы, переданные родителем, распределяются по соответствующим тегам `<ng-content>`. Если элемент соответствует селекторам сразу нескольких слотов, он спроецируется **только в первый совпавший слот по порядку его объявления в шаблоне**. В последующие слоты он не попадет. Если элемент не подошел ни под один селектор, он отправляется в слот по умолчанию (без атрибута `select`), если таковой объявлен в шаблоне компонента.
