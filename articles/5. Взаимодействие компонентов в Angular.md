В компонентной архитектуре Angular интерфейс строится в виде иерархического дерева. Для создания динамических связей между родительскими и дочерними элементами фреймворк предоставляет механизмы передачи данных сверху вниз (входные свойства), отправки событий снизу вверх (выходные свойства) и прямого программного доступа к дочерним узлам (шаблонные запросы).

---

## 1. Глубокое понимание концепции (Core Concept)

### Входные свойства (Inputs): Декоратор vs. Signal API
Для передачи данных от родителя к потомку Angular исторически использовал декораторы, но в современных версиях полностью перешел на реактивные сигналы (Signals).

| Критерий | Декоратор `@Input()` | Функция `input()` (Signal Input) |
| :--- | :--- | :--- |
| **Природа значения** | Обычное свойство класса (например, `string`) | Сигнал (`InputSignal<T>`), доступный по вызову `value()` |
| **Реактивное отслеживание** | Требует хуков `ngOnChanges` или setter-методов | Вычисляется декларативно через `computed` или `effect` |
| **Строгая типизация** | Возможны расхождения при инициализации по умолчанию | Гарантирует строгую безопасность типов из коробки |
| **Обязательность (Required)**| Требует ручной настройки `{ required: true }` | Задается через лаконичный метод `input.required<T>()` |

*Signal Inputs* кардинально упрощают архитектуру. Поскольку входное свойство становится сигналом, Angular автоматически отслеживает его изменения. Это исключает необходимость написания императивного кода в методе `ngOnChanges` для перевычисления зависимых состояний.

---

### Выходные свойства (Outputs): EventEmitter vs. Output API
Для уведомления родительского компонента о событиях внутри дочернего используется механизм генерации событий (Event Emitting).

1.  **Классический подход (`@Output`)**: Объявляется свойство класса с декоратором `@Output()`, инициализированное экземпляром `EventEmitter<T>` из пакета `@angular/core`. Данные отправляются вызовом `this.myOutput.emit(value)`.
2.  **Современный подход (`output()`)**: Инициализируется с помощью функции `output<T>()`. 
    *   *Важное отличие*: В отличие от `input()`, функция `output()` **не является сигналом**. Она возвращает специальный объект `OutputRef`, который не хранит состояние, а лишь предоставляет метод `.emit()`. Это сделано потому, что события по своей природе транзиенты (мимолетны) и не должны накапливать историю изменений. Использование `output()` избавляет от необходимости импортировать класс `EventEmitter`.

---

### Прямой доступ к дочерним элементам: ViewChild
Иногда родителю требуется напрямую вызвать метод дочернего компонента или прочитать его свойства. Для этого используются шаблонные запросы (Template Queries).

*   **Декоратор `@ViewChild(ChildComponent)`**: Ищет в шаблоне первый элемент, соответствующий указанному компоненту, директиве или локальной переменной шаблона (`#ref`). Результат становится доступен только после построения DOM-дерева шаблона (в хуке жизненного цикла `ngAfterViewInit`).
*   **Функция `viewChild(ChildComponent)`**: Современный реактивный аналог, возвращающий `Signal<ChildComponent | undefined>`. Преимущество заключается в том, что родительский компонент может декларативно реагировать на появление или уничтожение дочернего элемента в DOM-дереве с помощью эффектов, не дожидаясь срабатывания императивных хуков жизненного цикла.

## 2. Ментальная модель

### Поток данных и событий в иерархии компонентов

```text
               РОДИТЕЛЬСКИЙ КОМПОНЕНТ (Parent)
      ┌────────────────────────────────────────────────┐
      │  state = "Активен"                             │
      │  handleChildEvent(data) { ... }                │
      └───────┬────────────────────────────────▲───────┘
              │                                │
  [Входной поток]                              │ [Выходной поток]
  Передача данных                              │ Генерация событий
  через свойство:                              │ через подписку:
  [status]="state"                             │ (clicked)="handleChildEvent($event)"
              │                                │
              ▼                                │
      ┌────────────────────────────────────────┴───────┐
      │  status = input<string>()                      │
      │  clicked = output<string>()                    │
      │                                                │
      │               ДОЧЕРНИЙ КОМПОНЕНТ (Child)       │
      └────────────────────────────────────────────────┘
```

## 3. Практические примеры кода

### Плохой пример: Устаревшие декораторы, императивный ngOnChanges и утечки типов

Здесь изменение входного параметра отслеживается вручную через хук `ngOnChanges`. Это усложняет чтение кода, заставляет заводить приватные поля и снижает безопасность типизации.

```typescript
import { Component, Input, Output, EventEmitter, OnChanges, SimpleChanges } from '@angular/core';

@Component({
  selector: 'app-legacy-counter',
  standalone: true,
  template: `
    <div class="counter">
      <p>Счетчик: {{ count }} (Шаг: {{ step }})</p>
      <button (click)="increment()">Увеличить</button>
    </div>
  `
})
export class LegacyCounterComponent implements OnChanges {
  // Устаревшие декораторы входных свойств
  @Input() step: number = 1;
  @Input() initialValue: number = 0;

  // Избыточный EventEmitter
  @Output() countChange = new EventEmitter<number>();

  count: number = 0;

  // Сложное императивное отслеживание изменений входных параметров
  ngOnChanges(changes: SimpleChanges): void {
    if (changes['initialValue'] && changes['initialValue'].isFirstChange()) {
      this.count = this.initialValue;
    }
  }

  increment(): void {
    this.count += this.step;
    this.countChange.emit(this.count);
  }
}
```

### Хороший пример: Современный реактивный подход на Signals и Output API

Использование функций `input()` и `output()` делает код реактивным, избавляет от необходимости реализовывать хуки жизненного цикла и гарантирует строгую проверку типов на этапе сборки.

```typescript
// src/app/components/modern-counter/modern-counter.component.ts
import { Component, computed, input, output } from '@angular/core';

@Component({
  selector: 'app-modern-counter',
  standalone: true,
  template: `
    <div class="counter-box">
      <p>Значение: {{ count() }}</p>
      <p>Информация: {{ infoText() }}</p> <!-- Реактивное вычисляемое свойство -->
      <button (click)="increment()">Добавить шаг</button>
    </div>
  `,
  styles: [`
    .counter-box { padding: 1rem; border: 1px solid #eaeaea; }
  `]
})
export class ModernCounterComponent {
  // Обязательное входное свойство-сигнал
  initialValue = input.required<number>(); 
  
  // Опциональное входное свойство-сигнал со значением по умолчанию
  step = input<number>(1); 

  // Современный безопасный Output API
  valueChange = output<number>(); 

  // Локальное реактивное состояние на базе сигналов
  private internalCount = input<number>(0); // Значение по умолчанию

  // computed автоматически перевычислит значение, если изменятся зависимые сигналы
  readonly count = computed(() => this.initialValue() + this.internalCount());
  
  readonly infoText = computed(() => `Шаг прибавления равен: ${this.step()}`);

  private tempOffset = 0;

  increment(): void {
    this.tempOffset += this.step();
    // Имитируем обновление локального состояния
    this.valueChange.emit(this.initialValue() + this.tempOffset);
  }
}
```

```typescript
// src/app/components/parent/parent.component.ts
import { Component, viewChild, AfterViewInit } from '@angular/core';
import { ModernCounterComponent } from '../modern-counter/modern-counter.component';

@Component({
  selector: 'app-parent',
  standalone: true,
  imports: [ModernCounterComponent],
  template: `
    <main class="parent-container">
      <h2>Родительский компонент</h2>
      
      <!-- Передаем данные вниз и слушаем события снизу -->
      <app-modern-counter 
        [initialValue]="baseValue" 
        [step]="2"
        (valueChange)="onCounterChanged($event)" />
    </main>
  `
})
export class ParentComponent implements AfterViewInit {
  baseValue = 10;

  // Современный шаблонный запрос на сигналах (возвращает Signal<ModernCounterComponent | undefined>)
  readonly counterComponent = viewChild(ModernCounterComponent);

  onCounterChanged(newValue: number): void {
    console.log(`Получено новое значение от дочернего компонента: ${newValue}`);
  }

  ngAfterViewInit(): void {
    // Безопасное чтение значения из сигнала-запроса
    const childRef = this.counterComponent();
    if (childRef) {
      console.log(`Успешный доступ к дочернему компоненту. Текущий шаг: ${childRef.step()}`);
    }
  }
}
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем фундаментальное преимущество Signal-based входов (`input()`) перед классическими декораторами `@Input()`?
**Ответ:** 
Главное преимущество — глубокая интеграция с реактивной системой сигналов Angular:
1.  **Отсутствие хуков жизненного цикла**: Больше не нужно использовать `ngOnChanges` для отслеживания изменений свойств. Вы можете декларативно объединять входные параметры с другими сигналами через `computed()` или реагировать на изменения через `effect()`.
2.  **Повышение производительности (Change Detection)**: Традиционный CD Angular вынужден проверять весь компонент сверху вниз при любых изменениях. Сигналы позволяют Angular в будущем перейти на точечное (Zoneless) обновление шаблона исключительно в тех местах, где значение сигнала изменилось.
3.  **Безопасность инициализации**: С синтаксисом `input.required()` компилятор TypeScript гарантирует, что родитель не сможет забыть передать обязательный параметр, предотвращая ошибки типа `Cannot read properties of undefined` в рантайме.

### Вопрос 2: Является ли современный `output()` сигналом? Почему разработчики Angular приняли такое архитектурное решение?
**Ответ:** 
Нет, объект, возвращаемый функцией `output()`, **не является сигналом**.
Сигналы предназначены исключительно для хранения и декларативного обновления **состояний** во времени (они всегда имеют текущее значение и уведомляют об изменениях). 
События (Outputs) по своей природе являются мгновенными транзитными сообщениями (дискретными потоками данных), которые отправляются один раз и не сохраняют никакого постоянного состояния. Использование концепции сигналов для генерации событий привело бы к архитектурным ошибкам (например, ложным повторным срабатываниям событий при подписке). Именно поэтому `output()` возвращает `OutputRef`, работающий по классическому принципу подписок и генерации событий.

### Вопрос 3: Как решить проблему, когда `@ViewChild` возвращает `undefined` при попытке доступа к нему в методе `ngOnInit`?
**Ответ:** 
По умолчанию шаблонные запросы `@ViewChild` выполняются динамически (после полного построения DOM-дерева компонента), поэтому их результаты гарантированно доступны только начиная с хука жизненного цикла `ngAfterViewInit`. В `ngOnInit` они равны `undefined`.
Существует два способа решения:
1.  **Для декоратора `@ViewChild`**: Передать параметр `{ static: true }` в настройки декоратора: `@ViewChild(ChildComponent, { static: true })`. В этом случае Angular выполнит запрос до первой проверки изменений, сделав элемент доступным уже в `ngOnInit`. Однако это сработает только в том случае, если элемент не скрыт под структурными условиями (например, не находится внутри блоков `@if`).
2.  **Использовать Signal Query `viewChild()`**: Переход на современный `viewChild()` возвращает сигнал. Вы можете использовать метод `effect()`, который автоматически и безопасно среагирует на появление компонента в DOM-дереве в любой момент времени, избавляя от жесткой привязки к императивным методам `ngOnInit` или `ngAfterViewInit`.

### Вопрос 4: Как пересчитать локальное состояние дочернего компонента при изменении входного параметра `@Input` в классическом стиле и в стиле Signals?
**Ответ:** 
*   **В классическом стиле (декораторы)**: Приходится использовать грязные Setter-методы или реализовывать интерфейс `OnChanges`:
    ```typescript
    private _value!: string;
    @Input() set value(val: string) {
      this._value = val;
      this.derivedState = this.calculate(val); // Императивный пересчет
    }
    ```
*   **В стиле Signals**: Используется чистый декларативный подход с помощью функции `computed`:
    ```typescript
    value = input.required<string>(); // Сигнал
    derivedState = computed(() => this.calculate(this.value())); // Автоматический пересчет
    ```
    Второй вариант компактнее, не требует ручного создания приватных переменных хранения (`backing fields`) и автоматически гарантирует реактивность без рантайм-издержек.
