# 15. Работа с API и HttpClient в Angular

Организация взаимодействия с серверным API в Angular строится вокруг специализированного сервиса `HttpClient` и концепции реактивного программирования RxJS. Декларативный подход к выполнению запросов, гибкое проксирование трафика при разработке, управление конфигурациями сред и механизм промежуточного ПО (Interceptors) составляют основу надежной архитектуры сетевого взаимодействия.

---

## 1. Глубокое понимание концепции (Core Concept)

### Архитектура HttpClient, конфигурация proxy и environments

Для работы с HTTP-запросами в современных standalone-приложениях Angular используется вспомогательная функция `provideHttpClient()`, которая регистрирует необходимые провайдеры в конфигурации приложения `app.config.ts`.

#### Локальное проксирование (proxy.conf.json)
При локальной разработке фронтенд обычно запускается на `http://localhost:4200`, а бэкенд-сервер — на другом порту (например, `http://localhost:3000`). Разница в портах или доменах приводит к блокировкам CORS (Cross-Origin Resource Sharing) со стороны браузера. 

Для обхода этой проблемы на этапе разработки используется файл конфигурации проксирования `proxy.conf.json`. Он перенаправляет запросы с локального веб-сервера Angular на целевой адрес бэкенда.
Пример структуры `proxy.conf.json`:
```json
{
  "/api": {
    "target": "http://localhost:3000",
    "secure": false,
    "pathRewrite": {
      "^/api": ""
    },
    "changeOrigin": true
  }
}
```
Файл подключается при запуске через флаг CLI: `ng serve --proxy-config proxy.conf.json` или прописывается в `angular.json` в секции `serve.options.proxyConfig`. В production-окружении прокси не используется, так как скомпилированные статические файлы фронтенда обычно раздаются с того же домена и порта, на котором работает бэкенд.

#### Управление конфигурациями сред (Environments)
Для разграничения базовых URL-адресов API и других констант между средами разработки (Development), тестирования (Staging) и промышленной эксплуатации (Production) используется механизм сред Angular. 
Генерация файлов конфигурации выполняется командой CLI: `ng generate environments`.
В каталоге `src/environments/` создаются файлы `environment.ts` (по умолчанию для prod) и `environment.development.ts`. При сборке проекта Angular CLI производит автоматическую замену файлов (конфигурация `fileReplacements` в `angular.json`), подставляя нужный файл в зависимости от выбранного таргета сборки.

### Типизация запросов и опции HttpClient

Сервис `HttpClient` предоставляет строго типизированные методы (`get`, `post`, `put`, `delete` и др.), которые возвращают холодные RxJS Observable-потоки. Запрос отправляется в сеть только в момент подписки (`subscribe()`).

#### Типизация и форматы ответов
По умолчанию `HttpClient` ожидает, что сервер вернет данные в формате JSON и автоматически десериализует их. С помощью generic-типов можно явно типизировать возвращаемый ответ: `this.http.get<User[]>(...)`.

Если сервер возвращает не JSON-формат, поведение настраивается параметром `responseType`:
*   `'json'` (по умолчанию): парсит ответ в объект.
*   `'text'`: возвращает сырую строку.
*   `'blob'`: возвращает бинарные данные (Blob), например, для скачивания файлов.
*   `'arraybuffer'`: низкоуровневое представление бинарных данных.

#### Безопасная передача параметров
Для конструирования query-параметров (`?filter=all&page=1`) следует использовать иммутабельный класс `HttpParams` вместо ручной конкатенации строк. Это предотвращает синтаксические ошибки в URL и автоматически кодирует спецсимволы (URL-encoding).

---

### Перехватчики запросов (Interceptors)

Перехватчики функционируют как промежуточное программное обеспечение (middleware) для сетевых запросов. Они перехватывают исходящие `HttpRequest` и входящие `HttpResponse`, позволяя централизованно модифицировать их (например, прикреплять JWT-токен, обрабатывать ошибки или логировать время ответа).

В современном Angular приоритет отдается **функциональным перехватчикам** (`HttpInterceptorFn`) перед устаревшими перехватчиками на основе классов (`HttpInterceptor` через DI).
*   **Функциональные перехватчики** легче конфигурируются, не требуют регистрации глобальных мульти-провайдеров (`HTTP_INTERCEPTORS`) и обладают более предсказуемым поведением при ленивой загрузке модулей.
*   Регистрация осуществляется через функцию `withInterceptors` внутри конфигуратора `provideHttpClient`: `provideHttpClient(withInterceptors([authInterceptor, errorInterceptor]))`.

## 2. Ментальная модель

Цепочка прохождения HTTP-запроса от компонента к серверу и обратно через перехватчики и локальный прокси:

```text
 [ Angular Component ] ──► Вызов метода сервиса
        │
        ▼
 [ API Service ] ───────► Вызов HttpClient.get('/api/users')
        │
        ▼
 [ Functional Interceptor 1 ] (Auth: Добавляет заголовок Authorization)
        │
        ▼
 [ Functional Interceptor 2 ] (Logging: Логирует отправку запроса)
        │
        ▼
 [ Angular HttpBackend ]
        │
        ▼  [В режиме разработки: Локальный сервер localhost:4200]
 [ devServer Proxy ] ────► Переписывает путь '/api/users' в 'http://localhost:3000/users'
        │
        ▼  [Сетевой запрос по HTTP/HTTPS]
 [ Backend Server ] ────► Обрабатывает запрос и возвращает 200 OK со структурой JSON
        │
        ▼  [Обратный путь ответа]
 [ Functional Interceptor 2 ] (Logging: Вычисляет и логирует время выполнения запроса)
        │
        ▼
 [ Functional Interceptor 1 ] (Пропускает ответ дальше)
        │
        ▼
 [ API Service / Component ] ──► Получение типизированного JSON в subscribe()
```

## 3. Практические примеры кода

### Плохой пример: Ручная сборка параметров, классовые перехватчики и подписки без отписок

Ручная конкатенация строк параметров запроса приводит к уязвимостям и битым ссылкам. Использование старых классов-перехватчиков усложняет архитектуру, а отсутствие контроля жизненного цикла подписки ведет к утечкам памяти.

```typescript
// bad-api.service.ts
import { Injectable } from '@angular/core';
import { HttpClient, HttpHeaders } from '@angular/common/http';

@Injectable({ providedIn: 'root' })
export class BadApiService {
  constructor(private http: HttpClient) {}

  // Плохо: Ручная сборка URL-параметров (уязвимо к некорректному кодированию символов)
  // Плохо: Метод не типизирован, возвращает Observable<any> по умолчанию
  getUsers(filterValue: string, page: number) {
    const url = `/api/users?search=${filterValue}&page=${page}`;
    return this.http.get(url);
  }
}

// legacy-auth.interceptor.ts
import { Injectable } from '@angular/core';
import { HttpInterceptor, HttpRequest, HttpHandler, HttpEvent } from '@angular/common/http';
import { Observable } from 'rxjs';

@Injectable()
export class LegacyAuthInterceptor implements HttpInterceptor {
  // Плохо: Устаревший класс-перехватчик с многословной структурой DI
  intercept(req: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>> {
    const token = localStorage.getItem('token');
    if (token) {
      const cloned = req.clone({
        headers: req.headers.set('Authorization', `Bearer ${token}`)
      });
      return next.handle(cloned);
    }
    return next.handle(req);
  }
}
```

### Хороший пример: Современный типизированный сервис с функциональным перехватчиком

Использование декларативного реактивного сервиса, иммутабельного класса `HttpParams`, современного функционального перехватчика и RxJS-оператора `retry` для обработки временных сбоев связи.

```typescript
// auth.interceptor.ts
import { HttpInterceptorFn } from '@angular/common/http';

// Хорошо: Легковесный и производительный функциональный перехватчик
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = localStorage.getItem('token');
  
  if (token) {
    const modifiedReq = req.clone({
      headers: req.headers.set('Authorization', `Bearer ${token}`)
    });
    return next(modifiedReq);
  }
  
  return next(req);
};
```

```typescript
// app.config.ts
import { ApplicationConfig } from '@angular/core';
import { provideHttpClient, withInterceptors } from '@angular/common/http';
import { authInterceptor } from './auth.interceptor';

export const appConfig: ApplicationConfig = {
  providers: [
    // Хорошо: Конфигурация сетевого клиента с функциональными перехватчиками
    provideHttpClient(
      withInterceptors([authInterceptor])
    )
  ]
};
```

```typescript
// user-api.service.ts
import { Injectable, inject } from '@angular/core';
import { HttpClient, HttpParams } from '@angular/common/http';
import { Observable, retry, catchError, throwError } from 'rxjs';
import { environment } from '../environments/environment';

export interface User {
  id: string;
  name: string;
}

@Injectable({ providedIn: 'root' })
export class UserApiService {
  private http = inject(HttpClient);
  // Хорошо: Чтение базового URL из файла конфигурации среды
  private readonly baseUrl = environment.apiUrl; 

  getUsers(search: string, page: number): Observable<User[]> {
    // Хорошо: Использование HttpParams для безопасного и чистого построения query-параметров
    const params = new HttpParams()
      .set('search', search)
      .set('page', page.toString());

    // Хорошо: Явная строгая типизация ответа в generic-методе HttpClient.get
    return this.http.get<User[]>(`${this.baseUrl}/users`, { params }).pipe(
      retry({ count: 2, delay: 1000 }), // Автоматический перезапрос при сбое сети (2 попытки с интервалом 1с)
      catchError(error => {
        console.error('Сетевая ошибка:', error);
        return throwError(() => new Error('Не удалось загрузить пользователей. Попробуйте позже.'));
      })
    );
  }
}
```

## 4. Вопросы для собеседования

### Вопрос 1: В чем разница между функциональными перехватчиками (Functional Interceptors) и старыми перехватчиками на основе классов (DI Interceptors)? Каковы преимущества функционального подхода?
**Ответ:** 
*   **Интерцепторы на основе классов** требуют создания класса с декоратором `@Injectable`, реализующего интерфейс `HttpInterceptor`, и их обязательной громоздкой регистрации через мульти-провайдер `HTTP_INTERCEPTORS` в DI-контейнере.
*   **Функциональные перехватчики** представляют собой простые чистые функции типа `HttpInterceptorFn`. 
*   **Преимущества функционального подхода:**
    1.  **Простота и лаконичность кода:** меньше бойлерплейта, нет необходимости объявлять классы.
    2.  **Предсказуемость работы с DI:** функциональные интерцепторы без проблем работают с лениво загружаемыми провайдерами (Lazy Loading), тогда как старые DI-интерцепторы часто "терялись" или дублировались при использовании в дочерних модулях.
    3.  **Легковесность:** они лучше поддаются оптимизации компилятора (Tree Shaking), уменьшая итоговый размер бандла приложения.

### Вопрос 2: Зачем нужен файл `proxy.conf.json` при локальной разработке, как он настраивается и почему в нем нет необходимости на production-окружении?
**Ответ:** 
При локальной разработке фронтенд запускается на `localhost:4200`, а бэкенд на другом адресе, например `localhost:3000`. Попытка сделать прямой сетевой запрос с фронтенда на бэкенд будет заблокирована политикой браузера CORS (Same-Origin Policy). 
Файл `proxy.conf.json` настраивает локальный веб-сервер Webpack/Vite (запускаемый через `ng serve`) выступать в роли реверс-прокси. Он принимает запросы от фронтенда (например, на `/api/users`), перезаписывает заголовки и адрес хоста и перенаправляет запрос на бэкенд `localhost:3000` от своего имени (для браузера запрос выглядит беспрепятственным, так как он идет на тот же домен `localhost:4200`).
В production-окружении прокси не нужен, так как скомпилированные файлы фронтенда обычно раздаются веб-сервером (Nginx, Apache) с того же физического домена и порта, на котором хостится бэкенд, что исключает CORS-блокировки.

### Вопрос 3: Почему ручное конструирование строки параметров запроса вида `url?filter\=all` считается плохой практикой и как правильно использовать класс `HttpParams`?
**Ответ:** 
Ручное конструирование строк параметров запроса через конкатенацию (например, `url + "?q"\="+ query`) обладает следующими недостатками:
1.  **Проблема URL-encoding:** Если параметр содержит пробелы, кириллицу или спецсимволы (такие как амперсанд, знак вопроса, знак равенства `\=` и другие), ручная склейка приведет к некорректному URL-адресу и падению запроса. Придется оборачивать каждый параметр в функцию `encodeURIComponent()`.
2.  **Сложность поддержки:** При добавлении условных параметров (например, когда нужно добавить параметр в запрос, только если он не равен `null`) код превращается в трудночитаемую цепочку конкатенаций с ручной проверкой разделительных символов `?` и `&`.

Класс `HttpParams` является иммутабельным (каждый вызов метода `.set()` возвращает новый экземпляр класса). Он автоматически выполняет валидный URL-encoding для всех ключей и значений, предоставляет удобный декларативный интерфейс для добавления параметров по цепочке и полностью исключает синтаксические ошибки при формировании итогового URL-адреса.

### Вопрос 4: Как ведут себя перехватчики при обработке ошибок и как правильно реализовать автоматическую повторную отправку запроса (Retry) при временном сбое соединения на уровне RxJS?
**Ответ:** 
Каждый перехватчик возвращает `Observable<HttpEvent<any>>`. Если при выполнении сетевой операции происходит сбой, поток переходит в состояние ошибки. Мы можем перехватить эту ошибку внутри перехватчика или сервиса с помощью RxJS-оператора `catchError`.
Для автоматического восстановления соединения используется оператор `retry()`. Он принимает объект конфигурации, где можно указать количество повторных попыток отправки запроса (`count`) и задержку между попытками (`delay`). Если в процессе выполнения запроса возвращается ошибка (например, из-за кратковременной потери Wi-Fi связи), `retry` переподпишется на поток `HttpClient`, что заставит его инициировать новый физический запрос к серверу без участия пользователя. Если все попытки провалились, ошибка пробрасывается дальше через `throwError`.
